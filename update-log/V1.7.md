# TCM-Flow V1.7 更新说明：单书《景岳全书》生产 RAG 检索

日期：2026-06-18

## 1. 本版本目标

V1.7 将古籍 RAG 从实验目录收敛到项目运行链路中，并且只使用一本书：

- 原始文本：`637-景岳全书.txt`
- 书籍配置：`app/rag/config/ancient_books.yaml`
- 对外入口：`app/rag/retriever.py::retrieve_tcm_docs`
- 下游工具入口：`retrieve_tcm_knowledge`

本版本不是新增一轮实验，也不引入多书语料或外部知识库。它的重点是把《景岳全书》这一本书结构化、索引化，并接入现有项目的 RAG 调用路径，使问诊/咨询流程可以返回带证据位置的古籍检索结果。

## 2. 范围边界

V1.7 的生产 RAG 边界如下：

- 只读取 `637-景岳全书.txt`。
- 文本编码按 CP936/GBK 处理。
- 不再依赖旧版 `data/raw`、curated markdown 或实验目录输入。
- 不把 `experiments/rag_*` 的中间产物作为生产检索源。
- 检索结果只作为健康咨询和问诊信息整理的证据，不作为诊断或处方依据。
- 禁止根据检索结果生成方剂、药物、剂量、煎服法等治疗执行建议。

## 3. 《景岳全书》的结构化方式

原始 TXT 通过标签解析成层级结构：

```text
637-景岳全书.txt
  -> <目录> 行识别卷、篇、章路径
  -> <篇名> 行识别小节标题
  -> 正文清理属性前缀
  -> SelectedSection
  -> EvidenceParent
  -> RetrievalChunk
  -> rows.jsonl / bm25_tokens.jsonl / dense.npy
```

结构分层如下：

| 层级 | 说明 |
| --- | --- |
| Book | 当前只有 `jing_yue_quan_shu`，标题为《景岳全书》。 |
| Section | 由 `<目录>` 和 `<篇名>` 解析得到，保留 `volume`、`chapter`、`section`。 |
| EvidenceParent | 面向引用和最终回答的父证据，保存稳定 `parent_id`、症状标签、证据角色。 |
| RetrievalChunk | 面向召回和重排的检索单元，保存 `chunk_id`、`parent_id`、片段文本与索引字段。 |
| Index Row | 写入 BM25 token、dense embedding 和检索 manifest，用于运行时加载。 |

## 4. RAG 运行链路

从用户输入到输出结果，V1.7 的内部过程如下：

```text
用户问题
  -> detect_chief_symptom 识别主症
  -> rewrite_query 补充古籍别名
  -> retrieve_tcm_docs 保持旧公共入口
  -> get_production_engine 加载生产索引
  -> 症状标签过滤
  -> BM25 召回 + BGE-M3 dense 召回
  -> RRF 融合候选
  -> BGE reranker 重排
  -> parent 证据回收
  -> 生成 E1-E5 引用
  -> format_retrieval_results 输出可注入提示词的证据文本
```

例子：

```text
输入：用RAG检索帮我查一下眩晕是怎么回事
主症识别：眩晕
改写 query：原问题 + 眩晕 + 眩运 + 头眩
过滤范围：《景岳全书》中 symptom_tags 包含“眩晕”的父证据
输出：E1-E5，来源集中在 卷之十七理集·杂证谟 / 眩运
```

这样可以避免旧版“只按字面相似度乱召回”的问题，让现代口语症状先归一到项目支持的主症，再进入古籍证据空间。

## 6. 关于“眩晕”的当前检索结论

对“眩晕”类问题，当前系统可以稳定识别主症并召回《景岳全书》中的“眩运”章节证据。一次实际查询：

```text
query=眩晕伴耳鸣和乏力
chief_symptom=眩晕
status=ok
degraded=false
result_count=5
```

Top 证据集中在：

- 卷之十七理集·杂证谟 / 眩运 / 论证（共四条）
- 卷之十七理集·杂证谟 / 眩运 / 经义

因此，如果用户问“眩晕是怎么回事”，回答可以基于《景岳全书》的眩运证据，讨论上虚、下虚、痰、火、风等古籍病机线索；但不能把用户未提供的“持续 2 小时”“重心不稳”等情况写成事实，也不能直接给处方。

## 7. 与旧版 RAG 的差异

V1.7 保留 `retrieve_tcm_docs` 这个公共调用入口，但内部已经切换为 production ancient-books engine：

- 旧版轻量 BM25/markdown/documents 路径不再作为生产 RAG 来源。
- 查询改写和主症识别进入 `app/rag/ancient_books/query.py`。
- `app/rag/terms.py` 只保留术语抽取职责，不再作为 query rewrite/topic detect 的包装层。
- `app/rag/vector_store.py` 负责懒加载生产引擎，而不是维护旧版实验向量库。

这让外部调用方不用改 API，同时内部证据来源变得单一、可审计、可复现。

## 8. 当前限制

- 目前只覆盖《景岳全书》一书，不做跨书补证。
- 当前症状空间由 `ancient_books.yaml` 明确配置；未配置主症不会强行扩展。
- `胃脘痛` 返回不足证据，是为了避免用弱相关章节硬凑答案。
- 检索结果是古籍证据，不等价于现代医学诊断。
- 回答层必须保留就医提醒，尤其是急症、持续加重、神经系统异常、胸痛胸闷等情况。

## 9. 建议后续

V1.7 后续更适合做工程层加固，而不是立刻扩大实验：

- 在问诊 agent 中固定注入 `format_retrieval_results` 的回答约束。
- 对输出答案增加“不得越过证据”的检查。
- 增加一组面向真实用户问法的回归用例。
- 如果后续要扩书，应先新增配置和 manifest 校验，再讨论多书融合策略。
