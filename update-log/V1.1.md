# TCM-Flow V1.1 更新细节：Agent Trace 流式可视化与多 Task 动态拆解增强

## 一、版本主题

V1.1 的主题是：

```text
Agent Trace 流式可视化与多 Task 动态拆解增强
```

该版本是在 V1.0 Dynamic SubAgentExecutor 的基础上继续升级，重点解决两个问题：

```text
1. 如何让 Lead Agent 将复杂任务拆解为多个动态子任务；
2. 如何记录并展示 Agent 执行过程，使系统具备更强的可解释性。
```

V1.0 已经实现：

```text
Lead Agent
  ↓
task(description, expected_output)
  ↓
SubAgentExecutor
  ↓
DynamicSubAgent
  ↓
subagent_result
  ↓
Lead Agent 综合
```

V1.1 进一步升级为：

```text
Lead Agent
  ↓
Hybrid RAG 检索
  ↓
多 task 动态拆解
  ↓
多个 DynamicSubAgent 子任务执行
  ↓
agent_step 流式事件输出
  ↓
agent_trace 过程记录
  ↓
Guardrails 术语校验与答案重写
  ↓
final answer
```

---

## 二、V1.1 的设计目标

V1.1 的核心目标包括：

```text
1. 增强 Lead Agent 的任务拆解能力；
2. 支持复杂问题拆分为多个 task；
3. 记录工具调用开始、工具执行完成、子任务完成等关键过程；
4. 新增 agent_step SSE 事件，用于前端实时展示推理过程；
5. final 事件继续返回完整 agent_trace；
6. 保持 V0.7 Hybrid Search、V0.8 Guardrails、V1.0 DynamicSubAgentExecutor 的兼容性。
```

简单来说，V1.1 要让系统不仅“能做多智能体协同”，还要让前端和论文能够看到：

```text
系统为什么这样回答？
它先检索了什么？
它拆解了哪些子任务？
每个子任务做了什么？
最终答案有没有经过校验？
```

## 三、V1.1 与 V1.0 的区别

| 对比项        | V1.0                  | V1.1                                         |
| ---------- | --------------------- | -------------------------------------------- |
| task 调用    | 可以调用一个动态子任务           | 可以拆解为多个动态子任务                                 |
| 子任务粒度      | 一个 task 可能包含多个目标      | 每个 task 更倾向于只负责一个清晰目标                        |
| 过程记录       | final 中返回 agent_trace | 执行过程中实时推送 agent_step，同时 final 返回 agent_trace |
| 前端可视化      | 只能最终展示过程              | 可以实时展示执行过程                                   |
| Guardrails | final 后校验             | 保持 final 校验，同时记录校验过程                         |
| 论文价值       | 多智能体协同初步落地            | 推理过程可视化进一步增强                                 |

## 四、V1.1 的整体执行流程

V1.1 的整体执行链路如下：

```text
用户输入
  ↓
Lead Agent 判断任务复杂度
  ↓
调用 retrieve_tcm_knowledge 进行 Hybrid RAG 检索
  ↓
根据检索结果和用户需求拆解多个 task
  ↓
task 1：整理已知问诊信息和缺失信息
  ↓
task 2：基于检索证据总结可能相关因素
  ↓
task 3：整理日常调护建议和就医提醒
  ↓
Lead Agent 综合子任务结果
  ↓
Guardrails 检查 final answer 中的专业术语
  ↓
如果出现 unsupported_terms，则触发答案重写
  ↓
返回 final answer、validation、agent_trace
```

该流程体现了 Agentic RAG 的核心思想：

```text
不是简单地“检索 + 生成”，
而是“检索 + 动态规划 + 任务委派 + 结果综合 + 术语校验”。
```

## 五、核心更新一：多 Task 动态拆解规则增强

V1.0 中，Lead Agent 通常会把多个目标合并进一个 task，例如：

```text
整理问诊信息 + 分析可能因素 + 给出调护建议
```

这样虽然可以运行，但子任务粒度较粗，不利于过程可视化。

V1.1 在 Lead Agent Prompt 中加入了更明确的动态拆解规则：

```text
动态子任务拆解规则：
- 当 subagent_enabled=true，且用户问题包含多个步骤时，可以将复杂任务拆成多个 task 工具调用。
- 不要总是把所有目标塞进一个 task。
- 如果任务同时包含“问诊信息整理、证据分析、调护建议”，可以拆为：
  1. 一个 task 用于整理已知信息和缺失信息；
  2. 一个 task 用于基于检索证据总结可能相关因素；
  3. 一个 task 用于整理日常调护建议和风险提醒。
- 每个 task 的 description 应只包含一个清晰子目标。
- 每个 task 的 expected_output 应明确说明输出格式。
- 子任务结果只作为辅助分析，最终回答仍由你统一综合。
```

这使 Lead Agent 更倾向于把复杂中医问答任务拆解为多个独立的动态子任务。

## 六、核心更新二：新增 agent_step 流式事件

V1.1 新增 SSE 事件：

```text
event: agent_step
```

它用于实时推送 Agent 执行过程。

原来的事件主要是：

```text
event: values
event: final
event: end
```

V1.1 增加后，前端可以接收到：

```text
event: agent_step
event: values
event: final
event: end
```

其中 `agent_step` 可用于展示：

```text
1. Lead Agent 正在调用检索工具；
2. 中医知识检索完成；
3. Lead Agent 正在委派动态子任务；
4. DynamicSubAgent 子任务执行完成；
5. Guardrails 开始进行术语校验；
6. Guardrails 校验完成。
```

### 1. agent_step 增量事件提取

新增函数：

```python
def extract_trace_events_from_messages(
    messages: list[dict[str, Any]],
    emitted_keys: set[str],

```

该函数用于从 LangGraph 的 message 流中提取新发生的事件，包括：

为了避免 SSE 过程中重复推送同一个事件，函数使用：


## 八、核心更新四：worker.py 流式推送 agent_step

V1.1 修改：

```text
app/runtime/runs/worker.py
```

在 Agent 执行流中增加 `agent_step` 推送逻辑。

原流程：

```python
async for chunk in agent.astream(...):
    final_messages = [message_to_dict(m) for m in raw_messages]

    await bridge.publish(
        run_id,
        "values",
        {
            "messages": final_messages,
        },
    )
```

V1.1 增加：

```python
trace_events = extract_trace_events_from_messages(
    messages=final_messages,
    emitted_keys=emitted_trace_keys,
)

for trace_event in trace_events:
    await bridge.publish(
        run_id,
        "agent_step",
        trace_event,
    )
```

这样在执行过程中，一旦模型产生工具调用或工具返回结果，系统就能实时推送过程事件。

## 十二、V1.1 实际测试流程分析

本次测试中，系统完成了以下流程：

```text
1. Lead Agent 接收用户问题；

12. Lead Agent 综合检索结果与三个子任务结果生成回答；

14. 系统触发答案重写；
15. 重写后再次校验通过；
16. 返回 final answer 和 agent_trace。
```

## 十四、V1.1 当前存在的问题

### 问题一：信息不足时仍未严格调用 ask_clarification

测试结果中，最终回答仍包含大量追问：

```text
胃胀持续多久了？
食欲如何？
大便情况如何？
饮食习惯如何？
情绪压力如何？
是否有反酸、烧心？
```

但系统没有调用：

```text
ask_clarification
```

这说明 Lead Agent 仍倾向于在 final answer 中写追问，而不是触发澄清中断。

工程上，这会导致：

```text
1. 没有 event: clarification；
2. thread status 不会变成 waiting；
3. 前端无法识别当前轮是“等待用户补充信息”。
```

后续应继续强化 Prompt：

```text
如果你判断需要用户补充信息，必须调用 ask_clarification。
不要在 final answer 中写“请您先补充以下信息”这类正式追问。
如果只是初步分析，可以在结尾轻量提示“后续可补充持续时间、大便、饮食等信息”。
```

---

### 问题二：agent_trace 中可能包含未校验术语

虽然 final answer 已经经过 Guardrails 重写，但 `agent_trace` 中的子任务 summary 可能仍然保留原始子任务输出。

例如子任务中可能出现：

```text
胃气上逆
胃气壅滞
```

这些术语在 final answer 中被重写掉了，但如果前端直接展示完整 trace summary，用户仍可能看到未经最终校验的中间内容。

因此建议：

```text
agent_trace 面向用户展示时，只展示步骤，不展示完整医学分析内容。
```

例如将 summary 改成：

```text
动态子任务执行完成，结果已交由 Lead Agent 综合。
```

而不是直接展示子智能体原始输出。

---

### 问题三：agent_step 需要在 SSE 原始响应中确认

final JSON 中可以看到 `agent_trace`，但 `agent_step` 是 SSE 过程事件，不一定会出现在 final JSON 中。

需要在 Apifox、Postman 或前端 SSE 监听中查看原始流：

```text
event: agent_step
data: {...}

event: values
data: {...}

event: final
data: {...}

event: end
data: {...}
```

如果只看 final 响应体，就只能看到 `agent_trace`，看不到中间的 `agent_step`。

## 十五、V1.1 安全性优化建议

### 1. agent_trace 用户展示安全化

建议在 `trace_middleware.py` 中，将 task 的 summary 从原始内容：

```python
"summary": str(payload.get("content", ""))[:300]
```

改为：

```python
"summary": "动态子任务执行完成，结果已交由 Lead Agent 综合。"
```

这样前端展示的是过程，而不是未经校验的医学内容。

---

### 2. 区分开发者 Trace 和用户 Trace

后续可以设计两类 trace：

```text
developer_trace：
  展示完整 task_description、expected_output、subagent content，供调试使用。

user_trace：
  只展示安全摘要，例如“完成知识检索”“完成子任务分析”“完成术语校验”。
```

这样既方便开发调试，又避免向用户暴露未经校验的中间推理内容。

---

### 3. 强化 ask_clarification 边界

后续应继续优化 Lead Agent Prompt：

```text
如果需要用户补充信息：
  调用 ask_clarification

如果只是给出初步建议：
  final 中只轻量提示“后续可补充相关信息”
```

避免 final answer 同时承担“最终回答”和“正式追问”两个角色。

## 十七、V1.1 与 DeerFlow 思想的对应关系

| DeerFlow-like 设计思想 | TCM-Flow V1.1 实现                                    |
| ------------------ | --------------------------------------------------- |
| Lead Agent 总控      | `app/agents/lead_agent/agent.py`                    |

| 临时子智能体             | `DynamicSubAgent`                                   |
| 子任务执行器             | `SubAgentExecutor`                                  |
| 过程流式输出             | `event: agent_step`                                 |

| 安全校验               | `Guardrails`                                        |
| 检索增强               | Hybrid RAG                                          |

V1.1 的关键点是：

```text
不是提前写死“问诊 Agent、辨证 Agent、调护 Agent”，
而是由 Lead Agent 根据当前任务动态生成多个 task，
再由 DynamicSubAgent 临时执行局部任务。
```

这比固定流水线更接近 DeerFlow 的架构思想。

## 十九、后续版本建议

V1.2 可以考虑两个方向。

### 方向一：Clarification Controller 强化

目标：

```text
让 ask_clarification 成为真正的问诊中断机制。
```

重点解决：

```text
1. final answer 中不再写大量正式追问；
2. 信息不足时强制进入 clarification；
3. 每次 clarification 最多追问 3 个关键问题；
4. thread status 正确进入 waiting；
5. 用户补充信息后继续上一次问诊流程。
```

### 方向二：安全 Trace 双轨机制

目标：

```text
区分 user_trace 和 developer_trace。
```

设计：

```text
user_trace：
  给前端用户展示，只显示安全过程摘要。

developer_trace：
  给开发者调试，保留完整 task description、tool result 和 subagent content。
```

这样可以避免用户看到未经最终校验的中间医学术语。

## 二十、一句话总结

V1.1 的核心升级是：

```text
在 V1.0 动态子任务委派机制基础上，进一步实现多 task 动态拆解和 agent_step / agent_trace 过程记录，使系统具备更强的推理过程可视化能力。
```

该版本使系统从：

```text
Lead Agent + DynamicSubAgent 单次委派
```

升级为：

```text
Lead Agent 动态规划 + Hybrid RAG + 多 Task 子任务委派 + Agent Trace 可视化 + Guardrails 安全校验
```

这标志着 TCM-Flow 已经具备较完整的 Agentic RAG 与多智能体协同推理雏形。
