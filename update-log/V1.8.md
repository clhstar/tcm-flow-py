# TCM-Flow V1.8 更新说明：数据库持久化与单一路径 Database RAG

日期：2026-06-19

## 1. 本版本目标

V1.8 的目标是把项目从“本地内存状态 + 文件 artifacts 检索”的运行方式，升级为可持久化、可重建、可用数据库工具查看的工程版本。

本版本重点完成三件事：

1. 对话、运行记录和消息可以保存到 Postgres；
2. RAG 的向量数据、关键词数据和检索日志可以进入数据库/检索服务；
3. 线上 RAG 只保留一条运行路径：Postgres/pgvector + Elasticsearch。

V1.8 不新增新的论文实验，也不扩大古籍语料范围。它主要是工程化落地：让 V1.7 已经构建好的《景岳全书》生产 RAG artifacts 可以导入数据库，并通过统一的数据库检索链路对外服务。

## 2. 架构变化

V1.8 之后，线上服务的核心依赖变为：

```text
FastAPI / LangGraph runtime
  -> Postgres
       - app_threads
       - app_runs
       - app_messages
       - checkpoint 相关状态
       - rag_corpora / rag_sources / rag_parents / rag_chunks
       - rag_chunk_embeddings, vector(1024)
       - rag_bm25_tokens
       - rag_retrieval_logs
  -> Elasticsearch
       - tcm_rag_chunks_current alias
       - tcm_rag_chunks_v1_0_0 index
  -> retrieve_tcm_knowledge
       - database RAG engine
```

本地开发服务通过 `docker-compose.persistence.yml` 启动：

```powershell
docker compose -f docker-compose.persistence.yml up -d
```

当前端口约定：

```text
Postgres:      localhost:15432 -> container 5432
Elasticsearch: localhost:9200  -> container 9200
```

Postgres 使用 `pgvector/pgvector:pg16`，Elasticsearch 使用 `docker.elastic.co/elasticsearch/elasticsearch:8.15.0`。

## 3. 数据库表与持久化内容

### 3.1 应用运行数据

新增应用侧持久化表：

| 表 | 用途 |
| --- | --- |
| `app_threads` | 保存对话线程状态和 metadata |
| `app_runs` | 保存每次 run 的输入、上下文、状态和错误 |
| `app_messages` | 保存 thread 下的可展示消息和工具消息元数据 |
| `app_agent_trace_events` | 预留 agent trace 事件 |
| `app_validation_results` | 预留 guardrail/validation 结果 |

当 `CHECKPOINT_BACKEND=postgres` 时，thread/run store 会切换为 Postgres 版本。

### 3.2 RAG 数据

新增 RAG 持久化表：

| 表 | 用途 |
| --- | --- |
| `rag_corpora` | corpus 版本、manifest hash、模型信息和数量统计 |
| `rag_sources` | 书籍来源、源文件、hash 和编码 |
| `rag_sections` | 章节层级和症状标签 |
| `rag_parents` | Parent 证据上下文 |
| `rag_chunks` | Child 检索片段 |
| `rag_chunk_embeddings` | pgvector dense embedding |
| `rag_bm25_tokens` | 离线分词 token |
| `rag_retrieval_logs` | 检索请求、召回结果、融合结果和最终结果 |

当前 production corpus：

```text
corpus_id=ancient-books-v1.0.0
chunk_count=2288
embedding_dimension=1024
```

## 4. Database RAG 单一路径

V1.8 去掉线上 RAG 的双路径判断。服务运行时不再在 file/database 之间切换。

当前唯一线上链路：

```text
retrieve_tcm_knowledge
  -> app.rag.retriever.aretrieve_tcm_docs
  -> app.rag.vector_store.get_database_engine
  -> app.rag.database.engine.DatabaseRetrievalEngine
  -> pgvector dense search
  -> Elasticsearch keyword search
  -> RRF fusion
  -> reranker
  -> parent recovery
  -> formatted evidence output
```

`app/rag/vector_store.py` 现在只负责：

- 注册 database RAG engine；
- 清理 database RAG engine；
- 构建 `DatabaseRetrievalEngine`；
- 返回当前已注册的 database engine。

离线的 ancient-books CLI 仍然保留，只用于构建和检查 artifacts。

## 5. artifacts 导入流程

V1.8 保留 V1.7 的 ancient-books artifacts 作为数据来源：



## 6. 启动方式

完整本地启动流程：

```powershell
cd G:\work\tcm-flow
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
docker compose -f docker-compose.persistence.yml up -d
.\.venv\Scripts\python.exe -m app.rag.database.cli import-artifacts --corpus-dir data\rag\ancient_books\corpus --index-dir data\rag\ancient_books\index
.\.venv\Scripts\python.exe -m app.rag.database.cli rebuild-elasticsearch --corpus-dir data\rag\ancient_books\corpus --index-dir data\rag\ancient_books\index
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload --port 2027
```

`.env` 的核心配置：

```env
DATABASE_URL=postgresql://tcm:tcm@localhost:15432/tcm_flow
POSTGRES_POOL_SIZE=10
CHECKPOINT_BACKEND=postgres

ELASTICSEARCH_URL=http://localhost:9200
ELASTICSEARCH_RAG_INDEX_ALIAS=tcm_rag_chunks_current
ELASTICSEARCH_ANALYZER=standard
```


本地单节点 Elasticsearch 显示 `yellow` 通常是副本分片未分配，不代表主分片不可用。

## 8. 修复与加固

本版本同时修复了几个集成问题：

- `asyncpg`、`elasticsearch` 进入依赖清单；
- Elasticsearch Python client 固定在 8.x，避免和 ES 8.15 服务端出现兼容头版本不匹配；
- `.env` 由 `app.config.get_settings()` 加载；
- `/api/threads/{thread_id}/runs/stream` 中 `settings` 未定义的问题已修复；
- `app_messages` 会随 thread values 中的 messages 同步写入；
- artifacts loader 支持按 `rows.jsonl` 顺序重排 `chunks.jsonl`，避免 chunk 顺序不一致导致导入失败；
- `docs/RUNBOOK.md` 新增完整运行手册。

## 10. 当前限制

- V1.8 只完成生产 RAG 的数据库化和持久化，不改变 V1.7 的单书语料范围；
- Elasticsearch 当前使用 standard analyzer，中文分词质量后续可以继续优化；
- `rag_retrieval_logs` 的数据库写入结构已经准备好，后续还可以进一步把运行时检索日志和具体 run/thread 强绑定；
- 本地 Docker compose 适合开发和验证，生产部署还需要单独设计账号、密码、备份、监控和安全策略；
- `.env` 仍然是本地私有配置，不应提交。

## 11. 后续建议

V1.8 后更适合继续做工程闭环：

- 在真实 API 调用中验证 `app_messages`、`app_runs`、`rag_retrieval_logs` 的端到端落库；
- 给 Elasticsearch 增加更适合中文古籍检索的 analyzer；
- 增加数据库 doctor/smoke 命令，直接检查 Postgres 和 ES 的真实连接与数据量；
- 增加检索日志查询接口，让前端可以展示 evidence trace；
- 为 Docker persistence 增加更安全的生产部署模板。
