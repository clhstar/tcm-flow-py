# TCM-Flow V1.2 更新说明：Clarification Interrupt & Resume 澄清中断与恢复机制

## 一、版本主题

V1.2 的主题是：

```text
Clarification Interrupt & Resume
澄清中断与恢复机制
```

该版本是在 V1.1 的基础上继续升级，重点从“多智能体任务执行”转向“Agent Runtime 生命周期管理”。

V1.1 已经实现：

```text
Hybrid RAG
  ↓
多 task 动态拆解
  ↓
DynamicSubAgent 执行
  ↓
agent_step / agent_trace 可视化
  ↓
Guardrails 术语校验
```

V1.2 进一步实现：

```text
信息不足
  ↓
触发 clarification
  ↓
Run 中断
  ↓
Thread 进入 waiting 状态
  ↓
保存 pending_clarification
  ↓
用户补充信息
  ↓
同一 thread 自动 resume
  ↓
继续 RAG / task / guardrails / final
```

该版本的核心目标不是再增加新的中医专家 Agent，而是让系统具备更接近 DeerFlow 的 Runtime 能力。

---

## 二、V1.2 的设计思想

V1.2 学习的是 DeerFlow-like 的运行时架构思想：

```text
Agent 负责决策
Tool 表达意图
Runtime 捕获工具调用
Thread 保存状态
Run 管理生命周期
Worker 负责中断与恢复
SSE 推送过程事件
```

在普通问答系统中，模型可能会直接在 final answer 中写：

```text
为了更准确，请您补充……
```

但这种方式在工程上存在问题：

```text
1. 系统无法知道当前是否在等待用户补充；
2. 前端无法识别这是 clarification 状态；
3. Thread 状态不会进入 waiting；
4. 用户下一轮输入时，系统不知道这是对上一轮追问的回答。
```

因此 V1.2 将“追问”从普通文本升级为 Runtime 层的中断机制。

V1.2 的核心目标包括：


## 四、V1.2 的两条中断路径

V1.2 设计了两条 clarification 中断路径。

### 1. 主路径：ask_clarification 工具中断

理想流程是：

```text
Lead Agent 判断信息不足
  ↓
调用 ask_clarification
  ↓
Worker 捕获 ask_clarification tool result
  ↓
构造 clarification_payload
  ↓
保存 pending_clarification
  ↓
Thread 进入 waiting
  ↓
SSE 返回 event: clarification
```

这是最 DeerFlow-like 的方式。

因为 Agent 不是通过自然语言暗示“我想追问”，而是通过明确工具调用表达：

```text
我要中断当前 run，向用户追问信息。
```

---

### 2. 兜底路径：final_fallback 中断

实际测试中，模型有时没有主动调用 `ask_clarification`，而是在 final answer 中写正式追问。

例如：

```text
为了更准确地帮您分析，请先补充以下关键信息……
```

为了避免这种 final 被当作最终回答，V1.2 增加了 fallback 机制：

```text
模型输出 final
  ↓
Runtime 检测 final 是否包含正式追问
  ↓
如果是，则不返回 final
  ↓
转换为 clarification
  ↓
Thread 进入 waiting
```

这条路径不是主机制，而是保险机制。

## 六、pending_clarification 标准结构

V1.2 中，pending clarification 不再只是一个简单 question，而是结构化对象：

```json
{
  "type": "clarification",
  "run_id": "...",
  "question": "为了更准确地帮您分析，请先补充以下关键信息：\n1. ...\n2. ...\n3. ...",
  "required_fields": [
    "饮食情况",
    "食欲情况",
    "情绪压力"
  ],
  "source": "ask_clarification 或 final_fallback",
  "created_at": "...",
  "resume_count": 0
}
```

其中：

| 字段                | 说明                                          |
| ----------------- | ------------------------------------------- |
| `type`            | 固定为 clarification                           |
| `run_id`          | 触发澄清的 run                                   |
| `question`        | 面向用户展示的追问内容                                 |
| `required_fields` | 结构化缺失字段                                     |
| `source`          | 中断来源，可能是 ask_clarification 或 final_fallback |
| `created_at`      | 创建时间                                        |
| `resume_count`    | 恢复次数                                        |

## 七、Run / Thread 状态变化

V1.2 中一次 clarification 的状态变化如下：

```text
run.status = running
  ↓
触发 clarification
  ↓
run.status = waiting_clarification
thread.status = waiting
```

用户补充信息后：

```text
thread.values.pending_clarification 存在
  ↓
runtime 判断本轮是 resume
  ↓
run.status = running
  ↓
继续执行 Agent
  ↓
final 成功
  ↓
thread.status = idle
pending_clarification = None
```

这标志着系统开始具备真正的 Agent Runtime 生命周期管理能力。

## 八、V1.2 的 Resume 流程

当用户第二次或第三次在同一 thread 中输入补充信息时，系统会检查：

```python
thread.values.get("pending_clarification")
```

如果存在，则进入 resume 流程：

```text
用户输入补充信息
  ↓
start_run 检测 pending_clarification
  ↓
context.is_resume = true
  ↓
worker 构造 resume_record
  ↓
build_resume_input_data 包装用户输入
  ↓
发布 event: agent_step resume started
  ↓
继续执行 Agent
```

包装后的语义类似：

```text
用户正在回答上一轮澄清问题。

上一轮澄清问题：
...

需要补充的关键信息：
...

用户本轮补充：
...

请结合用户补充信息、历史对话和可用工具，继续完成中医健康咨询。
```

这样模型可以明确知道：

```text
当前输入不是一个全新问题，
而是对上一轮 clarification 的补充回答。
```

## 十二、当前存在的问题

### 问题一：agent_step 扫描了全量历史 messages

当前测试中，resume 后的 agent_step 不只展示了本轮新执行的步骤，还重新推送了历史 run 中的检索和 task 事件。

表现为：

```text
当前是第三轮 resume
但 agent_step 又推送了第一轮检索、第一轮 task、第二轮检索等旧事件。
```

原因是：

```text
trace_middleware 每次扫描的是全量 final_messages，
而不是本轮新增 messages。
```

这会导致前端过程展示混乱。

#### 修复方向

在 worker 中记录历史 messages 长度：

```python
previous_messages = thread_values.get("messages") or []
message_start_index = len(previous_messages)
```

提取 agent_step 时只扫描本轮新增消息：

```python
current_run_messages = final_messages[message_start_index:]

trace_events = extract_trace_events_from_messages(
    messages=current_run_messages,
    emitted_keys=emitted_trace_keys,
)
```

final 中的 agent_trace 也应只提取本轮新增：

```python
agent_trace = extract_agent_trace_from_messages(
    final_messages[message_start_index:]
)
```

---

### 问题二：当前主要依赖 final_fallback，而不是 ask_clarification 主动中断

测试中的 pending_clarification 来源为：

```json
"source": "final_fallback"
```

说明模型仍然倾向于先生成 final，再由 Runtime 兜底转换为 clarification。

这条链路能用，但不够理想。

理想的 DeerFlow-like 流程应该是：

```text
Lead Agent 判断信息不足
  ↓
直接调用 ask_clarification
  ↓
Worker 捕获工具
  ↓
立即中断
```

而不是：

```text
Lead Agent 先检索、task、生成 final
  ↓
final 中包含正式追问
  ↓
Runtime fallback 拦截
```

#### 修复方向

强化 Lead Agent Prompt：

```text
当用户只提供主诉和少量症状，且缺少持续时间、大便情况、反酸烧心、饮食情况、风险信号等关键信息时，应优先调用 ask_clarification。

不要先执行 retrieve_tcm_knowledge 或 task 后再追问。

如果用户明确要求“先按步骤分析”，但问诊信息明显不足，也应先调用 ask_clarification 收集关键缺失信息。
```

---

### 问题三：resume input 是否真正传入 Agent 需要确认

设计上，resume 时应传入包装后的 prompt：

```text
用户正在回答上一轮澄清问题。

上一轮澄清问题：
...

用户本轮补充：
...
```

但日志中展示的 human message 仍然是原始输入：

```text
过于油腻、食欲不振、压力稍大
```

这可能有两种情况：

```text
1. build_resume_input_data 没有真正传入 agent；
2. LangGraph checkpointer 展示的是合并后的历史 messages，而不是本轮真实输入。
```

#### 修复方向

临时增加 debug 事件：

```python
await bridge.publish(
    run_id,
    "agent_step",
    {
        "type": "resume_input",
        "status": "debug",
        "agent": "runtime",
        "summary": "resume input 已构造",
        "agent_input_data": agent_input_data,
    },
)
```

确认传入 Agent 的 input 是否已经包装。

### 问题四：agent_trace 中可能暴露未经校验的中间术语

子任务结果中可能出现：

```text
胃气壅滞
胃气上逆
气滞于中焦
```

这些术语虽然不会出现在最终 final 中，但如果前端直接展示 `agent_trace.summary`，用户仍然可能看到未经校验的中间分析内容。

#### 修复方向

将用户可见 trace 改为安全摘要：

```python
"summary": "动态子任务执行完成，结果已交由 Lead Agent 综合。"
```

完整子任务内容只保留给开发者调试，不直接展示给用户。

---

### 问题五：症状词与专业术语没有分级

本次 Guardrails 将：

```text
食欲不振
```

识别为 unsupported_terms。

但“食欲不振”更像用户提供的症状词，而不是证候、病机或治法术语。

如果所有症状词都必须来自 allowed_terms，后续会出现误判：

```text
用户说“口苦、乏力、失眠”
模型复述这些症状
却被判为 unsupported_terms
```

#### 修复方向

后续应将术语分级：

```text
症状词：允许来自用户输入或 RAG
证候词：必须来自 RAG allowed_terms
病机词：必须来自 RAG allowed_terms
治法词：必须来自 RAG allowed_terms
方药词：严格限制，默认不允许
```

## 十三、V1.2 的 DeerFlow-like 意义

这与 DeerFlow-like 架构中的核心思想一致：

```text
Agent 不是一次性问答函数，

```

## 十七、V1.3 建议方向

V1.2 完成后，V1.3 建议做：

```text
Run-local Trace & Safe Trace 双轨机制
```

核心目标：

```text
1. agent_step 只展示当前 run 的事件；
2. final agent_trace 只包含当前 run 事件；
3. user_trace 不暴露未校验术语；
4. developer_trace 保留完整调试信息；
5. Run Event Store 保存每一次 run 的事件记录。
```

这会进一步贴近 DeerFlow 的 Runtime / Event / Trace / Store 架构思想。

## 十八、一句话总结

V1.2 的核心升级是：

```text
将“追问”从普通 final 文本，升级为 Runtime 层可中断、可保存、可恢复的 Agent Run 生命周期能力。
```

该版本使系统从：

```text
Lead Agent + RAG + DynamicSubAgent
```

进一步升级为：

```text
Lead Agent + Runtime Interrupt + Thread Waiting + Resume Execution + Guardrails Final
```

这标志着 TCM-Flow 不再只是一个会调用工具的问答系统，而是开始具备 DeerFlow-like Agent Runtime 的关键能力。
