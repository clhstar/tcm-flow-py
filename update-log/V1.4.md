# TCM-Flow V1.4 更新说明：多智能体澄清与历史一致性

> 更新日期：2026-06-11  
> 版本主题：统一 Lead Agent 与 Dynamic SubAgent 的澄清流程，并保证 Guardrail 最终答案与 Checkpointer 历史一致。  
> 版本定位：在 V1.3 原生 Checkpointer 澄清流程基础上，补齐子 Agent 澄清、消息历史回写和结构化问题输出约束。

---

## 一、版本背景

V1.3 已经建立以下澄清流程：

```text
Lead Agent 调用 ask_clarification
  ↓
ClarificationMiddleware 生成 ToolMessage
  ↓
当前 LangGraph Run 到达 END
  ↓
Checkpointer 保存消息历史
  ↓
同一 Thread 的新 Run 自然继续
```

实际运行中仍存在三个问题：

```text
1. Guardrail 虽然重写了对外答案，但 Checkpointer 仍可能保留原始答案；
2. Dynamic SubAgent 发现信息不足时，只能返回普通文本，无法可靠中断当前 Run；
3. 模型传入带编号的问题时，最终可能出现“1. 1. 问题”。
```

V1.4 围绕这三个问题完善运行时闭环。

---

## 二、核心升级

V1.4 包含三项主要升级：

```text
1. Guardrail 重写后的答案覆盖 Checkpointer 中的最终 AIMessage；
2. Dynamic SubAgent 支持结构化 needs_clarification 状态；
3. Lead Agent 和子 Agent 通过 Prompt 统一约束澄清问题格式。
```

整体流程如下：

```text
用户输入
  ↓
Lead Agent
  ├─ 直接缺少信息：ask_clarification
  └─ 委派子任务：task
        ↓
    Dynamic SubAgent
        ├─ completed：返回 Lead Agent 综合
        └─ needs_clarification：Middleware 结束当前 Run
  ↓
Lead Agent 生成最终回答
  ↓
Guardrail 校验或重写
  ↓
如发生重写，使用相同 message id 覆盖 Checkpointer
  ↓
同步 ThreadStore 并返回 final
```

---

## 三、Guardrail 答案写回 Checkpointer

### 1. 原有问题

原流程为：

```text
Agent 生成原始回答
  ↓
LangGraph Checkpointer 保存原始 AIMessage
  ↓
Worker 调用 Guardrail
  ↓
对外返回 Guardrail 重写后的答案
```

这会造成历史不一致：

```text
SSE final：重写后的安全答案
ThreadStore conversation：重写后的安全答案
LangGraph Checkpointer：未经重写的原始答案
```

下一轮 Agent 从 Checkpointer 恢复时，仍可能看到被 Guardrail 删除或修改的内容。

### 2. 新增历史覆盖函数

`app/runtime/runs/worker.py` 新增：

```python
replace_final_ai_message_in_checkpoint()
```

处理步骤：

```text
1. 在本轮 messages 中找到最后一条不包含 tool_calls 的 AIMessage；
2. 读取该消息的 message id；
3. 使用相同 id 创建内容为 Guardrail final_text 的 AIMessage；
4. 调用 agent.aupdate_state() 更新 Checkpointer；
5. 重新调用 agent.aget_state() 读取完整消息；
6. 将更新后的 messages 写入 ThreadStore。
```

核心逻辑：

```python
await agent.aupdate_state(
    config,
    {
        "messages": [
            AIMessage(
                id=message_id,
                content=final_text,
            )
        ]
    },
)
```

只有 Guardrail 最终内容与 Agent 原始内容不一致时才执行覆盖。

### 3. 最终一致性

发生 Guardrail 重写后，以下数据使用同一最终答案：

```text
SSE final.content
ThreadStore conversation
ThreadStore messages
LangGraph Checkpointer messages
下一轮 Agent 上下文
```

## 四、Dynamic SubAgent 结构化返回

### 1. 输出协议与 JSON Mode

Dynamic SubAgent 使用 Pydantic 定义模型输出协议：

```python
class SubAgentOutput(BaseModel):
    status: Literal["completed", "needs_clarification"]
    clarification_questions: list[str]
    content: str
```

模型调用通过 DeepSeek 支持的 JSON Mode 执行：

```python
structured_model = model.with_structured_output(
    SubAgentOutput,
    method="json_mode",
)
```

这会让 API 使用 JSON Output，而不是仅依赖 Prompt 要求模型自行输出 JSON。

信息充分时：

```json
{
  "status": "completed",
  "clarification_questions": [],
  "content": "子任务结果"
}
```

信息不足时：

```json
{
  "status": "needs_clarification",
  "clarification_questions": [
    "腰痛持续多久了？",
    "是否伴有腿麻或下肢无力？"
  ],
  "content": "缺少持续时间和神经症状信息。"
}
```

约束如下：

```text
1. 只输出 JSON 对象；
2. 不使用 Markdown 代码块；
3. 澄清问题限制为 1 到 3 个；
4. content 用于向 Lead Agent 说明子任务结果或缺失信息。
```

### 2. DynamicSubAgentResult 扩展

`DynamicSubAgentResult` 新增：

```python
status: str = "completed"
clarification_questions: list[str] | None = None
```

`SubAgentExecutor` 返回的 `subagent_result` 新增：

```json
{
  "status": "needs_clarification",
  "needs_clarification": true,
  "clarification_questions": [
    "问题1",
    "问题2"
  ]
}
```

Trace 数据同步记录：

```text
needs_clarification
clarification_questions
```

### 3. 输出解析与业务校验

`app/subagents/dynamic_subagent.py` 新增：

```python
SubAgentOutput
parse_subagent_output()
```

解析流程：

```text
DeepSeek JSON Mode
  ↓
LangChain 结构化输出
  ↓
Pydantic 字段校验
  ↓
status 与 questions 业务关系校验
  ↓
转换为 DynamicSubAgentResult
```

结构化结果必须满足：

```text
1. 顶层必须是 JSON 对象；
2. status 只能是 completed 或 needs_clarification；
3. clarification_questions 必须是字符串数组；
4. needs_clarification 必须包含 1 到 3 个具体问题；
5. completed 状态不能包含澄清问题；
6. content 必须是字符串。
```

`parse_subagent_output()` 仍支持对 JSON 字符串和字典执行相同校验，便于测试和内部调用。

不再兼容自由文本、Markdown JSON 代码块，也不再根据“信息不足”等关键词猜测状态。JSON Mode 降低格式错误概率，Pydantic 与业务校验负责拒绝字段或状态不合法的结果。

## 五、子 Agent 澄清中断

### 1. Runtime 识别 task 结果

`extract_latest_clarification_question()` 支持两种澄清来源：

```text
1. ask_clarification ToolMessage；
2. task ToolMessage 中的 needs_clarification。
```

当 task 返回：

```json
{
  "type": "subagent_result",
  "needs_clarification": true,
  "clarification_questions": [
    "问题1",
    "问题2"
  ]
}
```

Runtime 会将问题格式化为统一的 clarification 文本。

### 2. Middleware 在下一次模型调用前结束 Run

`ClarificationMiddleware` 新增：

```python
@hook_config(can_jump_to=["end"])
def before_model(...):
    ...
```

普通子任务流程：

```text
task ToolMessage
  ↓
返回 Lead Agent
  ↓
Lead Agent 综合最终回答
```

需要澄清时：

```text
task ToolMessage
  ↓
ClarificationMiddleware.before_model
  ↓
jump_to = "end"
  ↓
不再执行下一次 Lead Agent 模型调用
```

这样可以避免子 Agent 已明确缺少信息，Lead Agent 却继续生成最终结论。

`completed` 状态不受影响，仍会返回 Lead Agent 综合。

### 3. Worker 等待 astream 自然结束

Worker 不再在 `async for chunk in agent.astream()` 循环内部立即返回。

新流程：

```text
循环中发现 clarification
  ↓
记录 clarification_to_emit
  ↓
继续等待 astream 自然结束
  ↓
LangGraph 完成 END 跳转和 Checkpointer 保存
  ↓
循环外发布 clarification
  ↓
Run 进入 waiting_clarification
```

这可以避免 Worker 已经返回，但 LangGraph 仍停留在待执行节点的情况。

## 六、澄清问题格式约束

澄清问题使用结构化数组传递，数组元素只包含问题正文：

```json
{
  "questions": [
    "腰痛持续多久了？",
    "是否伴有腿麻？",
    "是否有大小便异常？"
  ]
}
```

Lead Agent Prompt、Dynamic SubAgent Prompt、工具描述和 Pydantic 字段描述均增加以下规则：

```text
questions 或 clarification_questions 数组中的每个元素只写问题正文，
不要添加序号、项目符号或“问题1”等前缀。
```

正确与错误示例：

```json
{
  "正确": ["腰痛持续多久了？", "是否伴有腿麻？"],
  "错误": ["1. 腰痛持续多久了？", "问题2：是否伴有腿麻？"]
}
```

程序层仍负责：

```text
结构化 questions
  ↓
去除首尾空白
  ↓
按文本去重
  ↓
限制最多 3 个
  ↓
展示时统一添加 1. / 2. / 3.
```

程序不再识别或删除字符串内部已有的编号。如果模型违反 Prompt，仍可能出现重复编号；后续根据实际运行情况再决定是否恢复确定性清洗。

## 七、Trace 数据增强

`app/middlewares/trace_middleware.py` 调整：

```text
1. ask_clarification 的摘要从单个 question 改为 questions 数组；
2. task trace 记录 needs_clarification；
3. task trace 记录 clarification_questions。
```

示例：

```json
{
  "tool": "task",
  "needs_clarification": true,
  "clarification_questions": [
    "腰痛持续多久了？"
  ]
}
```

这些字段可用于前端过程展示、运行日志和后续问题定位。

## 九、V1.3 与 V1.4 对比

| 对比项 | V1.3 | V1.4 |
|---|---|---|
| 澄清来源 | Lead Agent `ask_clarification` | Lead Agent + Dynamic SubAgent |
| 问题格式 | 结构化数组 | Prompt 约束正文格式 + 基础去重 |
| task 缺少信息 | 依赖 Lead Agent 自行判断 | Middleware 在下一次模型调用前结束 Run |
| completed task | 返回 Lead Agent | 保持不变 |
| Guardrail 历史 | 只修改对外 final | 同步覆盖 Checkpointer AIMessage |
| clarification 发布 | 流循环中提前返回 | 等待 Agent 图自然结束后发布 |
| Trace | 主要记录 task 内容 | 增加澄清状态和问题数组 |

## 十一、RAG 改动回退说明

V1.4 开发期间曾尝试加入以下能力：

```text
1. BM25 标点、单字和零分结果过滤；
2. RAG 结果相关性判断；
3. relevant_count 和 has_relevant_evidence；
4. no_relevant_evidence 机器状态；
5. 无证据时的 Guardrail 固定回答。
```

实际测试发现，简单使用以下条件仍会保留无关文档：

```text
bm25_score > 0
keyword_hits > 0
topic_matches
```

“中医”“病机”“调护”等泛化词仍可能使胃病、失眠等文档被误判为腰痛相关证据。

因此，上述 RAG 改动已经从 V1.4 代码中回退，本版本不再声明具备：

```text
RAG 硬相关性门控
no_relevant_evidence 状态
无证据病机强制拦截
```

当前 RAG 行为保持 V1.3 版本，不属于 V1.4 的验收范围。

## 十二、已知限制

V1.4 当前仍存在以下限制：

```text
1. RAG 仍可能返回主题无关的知识文档；
2. 当前没有可靠的“无相关证据”机器状态；
3. Guardrail 术语检测可能漏掉未进入词表的医学判断；
4. 子 Agent 可能在正文中描述信息不足，却错误返回 completed；
5. 子 Agent 不遵守 JSON 协议时，当前子任务会直接失败；
6. completed 子任务中的内容仍需要 Lead Agent 做二次判断；
7. Prompt 不能绝对保证问题正文不携带编号，模型违约时仍可能重复编号；
8. InMemorySaver 在服务重启后不会保留 Checkpointer 历史。
```

---

## 十三、验收标准

V1.4 应满足：

```text
1. Guardrail 重写后覆盖相同 id 的最终 AIMessage；
2. Checkpointer 与 ThreadStore 保存相同最终答案；
3. 子 Agent 支持 completed 状态；
4. 子 Agent 支持 needs_clarification 状态；
5. 子 Agent 澄清问题限制为最多 3 个；
6. task 缺少信息时不再调用 Lead Agent 生成 final；
7. completed task 仍能继续由 Lead Agent 综合；
8. Lead Agent 与子 Agent Prompt 都明确禁止问题正文携带编号；
9. Middleware 对问题执行去空、去重和最多 3 条校验；
10. Worker 等待 astream 自然结束后再进入 waiting；
11. clarification 后 Checkpointer 不遗留待执行节点；
12. Trace 能记录子 Agent 的澄清状态和问题数组。
```

RAG 相关性与无证据拦截不属于本版本验收标准。

## 十四、后续建议

后续版本建议优先解决 RAG 相关性和 Agent 状态可靠性：

```text
1. 增加医疗泛词停用表；
2. 要求用户症状实体与文档主题实体存在交集；
3. 为 BM25 和向量距离分别建立经过评估的阈值；
4. 引入 query topic 与文档 metadata topic 的硬过滤；
5. 建立腰痛等真实查询的检索评测集；
6. 统计无证据率、误召回率和 TopK 准确率；
7. 扩充腰痛、颈肩痛等知识库主题；
8. 加强 completed 与 needs_clarification 的语义一致性校验；
9. 评估结构化输出失败时的有限重试机制；
10. 如果线上仍出现重复编号，再增加确定性编号清洗。
```

## 十五、一句话总结

V1.4 的核心升级是：

```text
让 Guardrail 最终答案真正进入历史，
让子 Agent 缺少信息时能够中断并进入统一澄清，
并通过统一 Prompt 约束结构化澄清问题的正文格式。
```

本版本重点解决多智能体运行状态和消息历史一致性，RAG 相关性优化已回退并留待后续版本重新设计。
