# TCM-Flow V1.9 更新说明：DeerFlow-like 后端流式协议收敛

日期：2026-06-21

## 1. 本版本目标

V1.9 的目标是把后端 SSE 流式协议收敛为 DeerFlow-like 结构，固定事件类型，避免前端或调试工具继续依赖项目自定义的临时事件。

本次更新只调整后端 stream 协议、worker 运行逻辑和相关后端测试，不修改前端代码。

更新后，后端允许发布的 SSE event 收敛为：

```text
metadata
messages
updates
values
final
error
end
```

以下旧事件不再作为 SSE event 发布：

```text
clarification
thinking
tool_result
answer_delta
agent_step
```

## 2. 协议变化

### 2.1 messages 成为唯一流式消息事件

`event: messages` 继续直接承载 LangGraph `stream_mode="messages"` 的原始输出，data 结构保持为：

```python
[
    serialized_message,
    metadata,
]
```

后端不再把思考内容、正式回复 token、工具调用请求、工具执行结果拆成额外 SSE event。它们都保留在 LangGraph message 自身结构中。

### 2.2 updates 承载过程状态

旧的 `agent_step` 发布点改为 `updates`。当 debug trace 或非 values/messages 的 LangGraph 过程事件需要对外发布时，统一走 `event: updates`。

这使过程状态和最终回复分离：

| 事件 | 职责 |
| --- | --- |
| `messages` | 原始 message 流 |
| `updates` | node、middleware、trace、guardrail 等过程状态 |
| `final` | 本轮最终业务结果 |

### 2.3 clarification 改为 final

澄清问题不再使用独立的 `event: clarification`。当本轮需要用户补充信息时，worker 会保存 thread values，然后发布：

```text
event: final
status: need_clarification
```

`final` payload 仍使用统一业务响应结构：

```python
{
    "thread_id": "...",
    "run_id": "...",
    "status": "need_clarification",
    "assistant_message": "...",
    "pending_clarification": [...],
    "references": [],
}
```

线程和 run 状态仍进入等待澄清：

```text
run.status = waiting_clarification
thread.status = waiting
```

## 3. Worker 流式逻辑

后端内部固定使用 list 形式的 stream mode，并始终补齐：

```python
["messages", "values"]
```

如果请求方额外要求 `updates`，worker 会保留该模式。这样 LangGraph 多模式输出始终按 `(stream_event, chunk)` 处理，不再回退为单一字符串模式。

对外发布 `values` 由以下条件控制：

```python
publish_values = "values" in requested_stream_modes or debug_events
```

因此：

1. 后端内部永远订阅 `values`，用于保存 final_messages、提取最终回复和处理澄清；
2. 默认请求只对外发布 `messages`、`final`、`error`、`end` 等必要事件；
3. 请求方显式包含 `values` 或开启 `debug_events` 时，才对外发布完整 values 快照。

## 4. 序列化变化

新增统一 message serializer，保留 LangChain / LangGraph message 的关键协议字段：

```text
type
content
id
name
additional_kwargs
response_metadata
tool_calls
tool_call_chunks
invalid_tool_calls
tool_call_id
usage_metadata
chunk_position
status
artifact
```

这避免 reasoning_content、tool_call_chunks、ToolMessage.tool_call_id 和 usage_metadata 在 SSE 或 values 快照中丢失。

## 5. 最终回复提取

最终 assistant_message 继续从完整 messages 中反向提取，但会跳过带 `tool_calls` 的中间 AI 消息：

```python
def extract_final_ai_text(messages):
    for msg in reversed(messages):
        if msg.get("type") == "ai" and msg.get("content") and not msg.get("tool_calls"):
            return str(msg["content"])
    return ""
```

这保证工具调用请求不会被误当作最终可见回复。

## 6. 请求默认值

`RunCreateRequest.stream_mode` 默认值从 `["values"]` 调整为：

```python
["messages"]
```

服务层会把请求体中的 stream_mode 转发到 worker context。即使请求方只请求 `values`，worker 内部仍会补齐 `messages`，保证后端流式处理结构稳定。

## 8. 结果

V1.9 之后，后端流式协议边界更清晰：

1. `messages` 负责原始 LangGraph message 流；
2. `updates` 负责过程状态；
3. `values` 负责完整 state 快照，并默认只供后端内部使用；
4. `final` 负责最终可见业务结果，包括需要澄清的场景；
5. 不再新增 clarification/thinking/tool_result/answer_delta/agent_step 等额外 SSE event。
