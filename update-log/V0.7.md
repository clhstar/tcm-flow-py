# TCM-Flow V0.7 更新文档：Hybrid Search 混合检索

> 版本主题：从单一向量检索升级为“向量语义检索 + BM25 关键词检索 + 结果融合 + 重排”的混合检索增强版本。  
> 版本定位：V0.7 是系统中“混合检索增强”能力真正落地的关键版本，为后续论文中的混合检索模块、消融实验和检索效果评估打基础。

---

## 一、V0.7 版本目标

V0.6 已经实现了基础 RAG 优化能力：

```text
Query Rewrite
  ↓
Chroma 向量检索
  ↓
规则 Rerank
  ↓
Allowed Terms
  ↓
Retrieval Log
```

但 V0.6 仍然主要依赖单一路径的向量检索。  
V0.7 的目标是升级为真正的混合检索：

```text
Query Rewrite
  ↓
并行检索
  ├─ Dense Vector Search：向量语义检索
  └─ BM25 Keyword Search：关键词检索
  ↓
Result Fusion：结果融合
  ↓
Final Rerank：融合重排
  ↓
Allowed Terms：术语约束
  ↓
Evidence Output：证据输出
```

V0.7 的核心目标可以总结为：

```text
让系统同时具备“语义相似召回”和“专业术语精确匹配”的能力。
```

---

## 二、为什么需要 V0.7？

### 1. 单一向量检索的不足

V0.6 使用 Chroma + MiniMax Embedding 进行向量检索，适合处理语义相近的问题，例如：

```text
用户说：肚子胀胀的，吃完饭不舒服
知识库写：胃胀、胃脘胀满、饭后加重
```

向量检索可以通过语义相似性找到相关内容。

但向量检索也存在问题：

```text
1. 对专业术语的精确匹配不一定稳定
2. 对短词、证候名、病机名可能召回不准
3. 用户明确提到术语时，关键词检索反而更直接
4. 检索结果排序容易受语义噪声影响
```

例如：

```text
脾胃虚弱
气机不畅
心肾不交
肝郁化火
嗳气
大便不成形
```

这些中医术语需要较强的精确匹配能力，仅靠向量检索不够稳。

---

### 2. BM25 的补充价值

BM25 关键词检索适合解决：

```text
1. 专业术语精确匹配
2. 症状词精确命中
3. 章节内容中关键词出现频率较高的情况
4. 用户明确询问某个术语定义或表现
```

例如用户问：

```text
脾胃虚弱是什么意思？
```

BM25 可以直接命中包含“脾胃虚弱”的 chunk。

---

### 3. 混合检索更适合中医问答

中医问答场景同时存在两类表达：

```text
1. 用户口语化表达：
   胃里胀胀的、老打嗝、睡不着、早上腰酸

2. 中医专业化表达：
   胃胀、嗳气、脾胃虚弱、气机不畅、心肾不交
```

因此，V0.7 使用混合检索：

```text
向量检索：解决口语表达与专业知识之间的语义匹配
BM25 检索：解决症状、证候、病机等术语的精确匹配
```

这也更符合论文题目中的“混合检索增强”。

---

## 三、V0.7 整体架构

V0.7 的检索流程如下：

```text
用户输入
  ↓
Lead Agent 判断需要检索
  ↓
调用 retrieve_tcm_knowledge
  ↓
大模型自动生成 query
  ↓
rewrite_tcm_query 规则扩展
  ↓
并行检索
  ├─ vector_search：Chroma 向量检索
  └─ bm25_search：BM25 关键词检索
  ↓
merge_results：合并去重
  ↓
calculate_fusion_score：融合打分
  ↓
rerank_merged_results：最终重排
  ↓
collect_allowed_terms：提取允许术语
  ↓
返回 evidence + source + allowed_terms
  ↓
Agent 基于证据回答或调用 ask_clarification
```

## 五、V0.7 检索模式说明

### 1. Hybrid 模式

默认模式：

```text
mode = "hybrid"
```

流程：

```text
Query Rewrite
  ↓
Chroma 向量检索
  ↓
BM25 关键词检索
  ↓
合并去重
  ↓
融合重排
```

适合大多数中医问答场景。

典型问题：

```text
我最近胃胀，吃完饭更明显，怎么办？
晨起腰酸背痛，从中医看可能是什么原因？
失眠多梦，平时可以怎么调护？
```

---

### 2. Vector 模式

```text
mode = "vector"
```

只使用向量检索。

适合：

```text
1. 用户表达非常口语化
2. 没有明确专业术语
3. 需要语义相似匹配
```

示例：

```text
我肚子总是胀胀的，吃完饭不舒服
```

---

### 3. Keyword 模式

```text
mode = "keyword"
```

只使用 BM25 关键词检索。

适合：

```text
1. 用户明确询问某个专业术语
2. 需要精确匹配知识库原文
3. 用户问题中有明确症状词、证候名、病机名
```

示例：

```text
脾胃虚弱是什么意思？
气机不畅有哪些表现？
心肾不交和失眠有什么关系？
```

---

### 4. 推荐策略

正式问答中建议：

```text
默认强制 hybrid
vector / keyword 仅用于调试和实验对比
```

推荐设计：

```text
正式 Agent 工具：
retrieve_tcm_knowledge(query) 内部固定 mode="hybrid"

实验调试接口：
/api/rag/search?query=胃胀&mode=vector
/api/rag/search?query=胃胀&mode=keyword
/api/rag/search?query=胃胀&mode=hybrid
```

这样既保证正式系统稳定，又方便论文做消融实验。

## 六、V0.7 融合策略

V0.7 使用简化版融合打分：

```text
fusion_score =
  vector_rrf * 100
  + bm25_rrf * 100
  + overlap_bonus
  + keyword_hits * 2
  + topic_bonus
  + section_bonus
```

其中：

```text
vector_rrf：向量检索排名倒数得分
bm25_rrf：BM25 检索排名倒数得分
overlap_bonus：同一 chunk 同时被 vector 和 BM25 命中时加分
keyword_hits：中医术语命中数量
topic_bonus：topic 与 query 主题匹配时加分
section_bonus：问诊、调护、危险信号等章节加分
```

### 1. RRF 排名融合

V0.7 使用了简化 RRF 思路：

```text
vector_rrf = 1 / (60 + vector_rank)
bm25_rrf = 1 / (60 + bm25_rank)
```

排名越靠前，得分越高。

---

### 2. 双路命中加分

如果同一个 chunk 同时被两路检索命中：

```text
retrieval_sources = ["vector", "bm25"]
```

则说明它同时具有语义相关性和关键词相关性，因此加：

```text
overlap_bonus = 3
```

---

### 3. 术语命中加分

如果 chunk 中包含 query 中的中医术语，则增加分数：

```text
keyword_hits * 2
```

例如 query 中有：

```text
胃胀、嗳气、反酸、大便不成形
```

chunk 中命中越多，排序越靠前。

---

### 4. 主题和章节加分

如果识别到用户主题是“胃胀”，且 chunk 的 topic 也是“胃胀”，则加分。

如果章节是：

```text
问诊
调护
危险信号
概述
```

也会获得不同程度的加分。

## 十、V0.7 对论文的价值

V0.7 可以支撑论文中的“混合检索增强模块”。

可以这样描述：

```text
系统在知识检索阶段采用混合检索策略，将基于 MiniMax Embedding 与 Chroma 的语义向量检索和基于 BM25 的关键词检索相结合。向量检索用于捕捉用户口语化表达与中医知识之间的语义相关性，BM25 关键词检索用于提高症状、证候、病机等中医专业术语的精确匹配能力。随后，系统对两路检索结果进行融合与重排序，并结合主题、章节和术语命中情况筛选最终证据片段，从而提升检索结果的相关性、稳定性和可追溯性。
```

## 十二、V0.7 仍然存在的不足

V0.7 虽然实现了混合检索，但仍然存在以下不足：

```text
1. BM25 索引目前是运行时从 data/raw 构建，尚未持久化
2. 融合策略是规则版，不是训练好的学习排序模型
3. Rerank 仍然是规则重排，不是 BGE Reranker 或 Cross-Encoder Reranker
4. Allowed Terms 仍然只是提示约束，没有强制答案校验
5. 检索效果还没有用标准指标进行量化评估
6. 知识库内容规模仍然较小
```
