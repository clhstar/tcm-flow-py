# TCM-Flow V0.8 更新细节：术语校验与答案重写机制

## 一、版本主题

V0.8 的主题是：

```text
术语校验与答案重写机制
```

该版本是在 V0.7 混合检索增强的基础上，进一步增加 **Guardrails 生成校验层**，用于检查大模型最终回答中的中医专业术语是否具有检索证据支持。

V0.7 解决的是：

```text
知识从哪里来，检索结果是否更相关
```

V0.8 解决的是：

```text
最终回答有没有乱用没有知识库依据的中医术语
```

因此，V0.8 是从“检索增强”走向“证据约束生成”的关键版本。

---

## 二、V0.8 的开发背景

在 V0.7 中，系统已经实现了：

```text
Query Rewrite
  ↓
向量检索 + BM25 关键词检索
  ↓
结果融合
  ↓
Rerank
  ↓
Allowed Terms
  ↓
Agent 生成回答
```

其中 `Allowed Terms` 用于告诉大模型：

```text
回答中的中医专业术语应优先来自允许使用的专业术语
```

但是，这种约束仍然主要依赖 Prompt。
大模型可能仍然会在最终回答中主动引入未被检索结果支持的术语。

例如，检索结果只支持：

```text
胃胀
饮食不节
情志不畅
脾胃虚弱
气机不畅
嗳气
恶心
```

但模型回答中可能生成：

```text
可能与肝胃不和、寒湿困脾有关。
```

如果 `肝胃不和`、`寒湿困脾` 没有出现在本轮检索证据或 allowed_terms 中，那么它们就属于无依据术语，存在术语幻觉风险。

因此，V0.8 的目标是：

```text
回答生成后，对回答中的中医术语进行校验；
发现无依据术语后，触发答案重写。
```

---

## 三、V0.8 的核心目标

V0.8 的核心目标可以概括为：

```text
将 V0.7 的 Prompt 术语约束，升级为工程级术语校验机制。
```

具体包括：

```text
1. 从最终回答中抽取中医专业术语
2. 从最近一次 RAG 检索结果中解析 allowed_terms
3. 对 answer_terms 和 allowed_terms 做一致性比对
4. 找出 unsupported_terms
5. 如果存在 unsupported_terms，则调用 LLM 重写答案
6. 重写后再次校验
7. 在 final 事件中返回 validation 信息
```

整体流程如下：

```text
Agent 生成 final answer
  ↓
抽取回答中的中医术语 answer_terms
  ↓
解析检索工具返回的 allowed_terms
  ↓
比对 answer_terms 与 allowed_terms
  ↓
发现 unsupported_terms
  ↓
调用 answer_rewriter 重写答案
  ↓
再次执行术语校验
  ↓
返回最终回答 + validation 信息
```

## 四、V0.8 新增目录结构

V0.8 新增了 Guardrails 目录：

```text
app/
├── guardrails/
│   ├── __init__.py
│   ├── term_checker.py
│   ├── answer_validator.py
│   └── answer_rewriter.py
```

各文件职责如下：

| 文件                    | 作用                         |
| --------------------- | -------------------------- |
| `term_checker.py`     | 基于中医术语词典，从回答中抽取专业术语        |
| `answer_validator.py` | 校验回答中的术语是否来自 allowed_terms |
| `answer_rewriter.py`  | 当校验不通过时，调用大模型重写答案          |
| `__init__.py`         | Guardrails 模块初始化           |

## 五、核心模块一：术语抽取 `term_checker.py`

### 1. 功能定位

`term_checker.py` 用于从最终回答中抽取中医专业术语。

该模块当前采用的是：

```text
中医术语词典 + 完整字符串匹配
```

而不是 jieba 分词，也不是大模型抽取。

原因是：

```text
术语校验需要精确识别完整中医术语，不希望被普通分词切碎。
```

例如回答中有：

```text
可能与肝胃不和、寒湿困脾有关。
```

系统不是把它切成：

```text
肝胃 / 不和
寒湿 / 困脾
```

而是直接检查：

```text
“肝胃不和” 是否完整出现在文本中
“寒湿困脾” 是否完整出现在文本中
```

因此能精准抽取：

```json
["肝胃不和", "寒湿困脾"]
```

---

### 2. 术语来源

V0.8 的术语来源包括两部分：

```text
1. app/rag/terms.py 中维护的 TCM_TERMS
2. app/guardrails/term_checker.py 中新增的 EXTRA_TCM_TERMS
```

其中 `EXTRA_TCM_TERMS` 主要用于检测模型是否乱引入常见证候、病机或治法术语。

例如：

```text
肝胃不和
食积内停
寒湿困脾
脾虚湿困
胃气上逆
肝郁气滞
肝气犯胃
湿热中阻
胃阴不足
脾阳虚
肾阳虚
肾阴虚
气血不足
气滞血瘀
痰湿内阻
健脾和胃
理气和中
疏肝理气
消食导滞
温中散寒
清热化湿
滋阴养胃
```

这些术语不一定都允许模型使用，但它们需要被系统识别出来，用于判断是否存在无依据术语。

---

### 3. 长词优先策略

术语抽取时使用：

```python
sorted(terms, key=len, reverse=True)
```

即长词优先。

这样做的目的是减少短词对长词的干扰。

例如：

```text
胃脘胀满
胃胀
```

系统会优先检查更长的：

```text
胃脘胀满
```

再检查：

```text
胃胀
```

当前版本采用词典匹配，优点是简单、稳定、可控；缺点是只能识别词典中已有的术语，后续需要继续扩展中医术语表。

## 六、核心模块二：答案校验 `answer_validator.py`

### 1. 功能定位

`answer_validator.py` 是 V0.8 的校验入口。

核心逻辑是：

```text
answer_terms = 从最终回答中抽取出的中医术语
allowed_terms = 本轮 RAG 检索结果允许使用的术语

unsupported_terms = answer_terms - allowed_terms
```

如果 `unsupported_terms` 为空，说明回答通过术语校验。

如果 `unsupported_terms` 不为空，说明回答中出现了检索证据未支持的中医术语，需要触发重写。

---

### 2. 校验结果结构

校验函数返回结构如下：

```json
{
  "passed": false,
  "reason": "unsupported_terms_found",
  "answer_terms": [
    "饮食不节",
    "气机不畅",
    "肝胃不和",
    "寒湿困脾",
    "胃胀",
    "嗳气",
    "恶心"
  ],
  "allowed_terms": [
    "胃胀",
    "嗳气",
    "反酸",
    "烧心",
    "恶心",
    "便秘",
    "腹泻",
    "大便不成形",
    "饮食不节",
    "情志不畅",
    "脾胃虚弱",
    "气机不畅",
    "呕吐",
    "黑便",
    "消瘦"
  ],
  "unsupported_terms": [
    "肝胃不和",
    "寒湿困脾"
  ]
}
```

这说明模型回答中出现了：

```text
肝胃不和
寒湿困脾
```

但这两个术语不在 allowed_terms 中，因此判定为无依据术语。

---

### 3. 没有 allowed_terms 时的处理

如果本轮没有触发 RAG 检索，或者无法解析出 allowed_terms，则暂时跳过严格校验：

```json
{
  "passed": true,
  "reason": "no_allowed_terms_skip_validation"
}
```

这样可以避免普通闲聊或非中医知识检索场景被误判。

## 七、核心模块三：答案重写 `answer_rewriter.py`

### 1. 功能定位

当 `answer_validator.py` 检测到 `unsupported_terms` 时，系统会调用 `answer_rewriter.py`，让大模型基于检索证据和 allowed_terms 重写答案。

重写目标是：

```text
1. 删除没有证据支持的中医术语
2. 不引入新的证候、病机、治法或方剂术语
3. 保留原答案中合理的健康建议
4. 使用谨慎表达
5. 不直接下诊断
6. 不开处方和剂量
```

---

### 2. 重写 Prompt 约束

V0.8 中，重写 Prompt 明确要求：

```text
你是一个中医健康问答系统的答案重写助手。

你的任务是直接生成“面向用户展示的最终回答”。

严格要求：
1. 只能输出重写后的最终回答正文。
2. 不要说“我已重写”“重写后的答案如下”“根据您提供的证据”等过程性话语。
3. 不要输出分隔线，例如“---”。
4. 不要解释你是如何重写的。
5. 删除或替换没有证据支持的中医专业术语。
6. 不要引入新的证候、病机、治法或方剂术语。
7. 不要直接下诊断。
8. 不要开处方或给剂量。
9. 表达要谨慎，可以使用“可能相关”“可先从……角度理解”“目前依据有限”等说法。
10. 保留必要的日常调护建议和线下就医提醒。
```

---

### 3. 元话术清理

在测试中发现，重写模型可能会输出：

```text
好的，根据您提供的检索证据和允许使用的专业术语，我已对原答案进行重写。重写后的答案如下：
---
```

这类内容属于系统内部过程性话语，不应该展示给用户。

因此 V0.8 增加了 `clean_rewritten_answer()` 兜底清理函数，用于删除：

```text
好的，重写后的答案如下：
以下是重写后的答案：
我已对原答案进行重写。
---
```

最终用户只看到真正的回答正文。

## 八、`worker.py` 核心改造

V0.8 的主要集成点在 `worker.py`。

原来的 final 流程是：

```text
Agent 执行完成
  ↓
extract_final_ai_text
  ↓
append_visible_messages
  ↓
publish final
```

V0.8 改造后变为：

```text
Agent 执行完成
  ↓
extract_final_ai_text
  ↓
extract_allowed_terms_from_messages
  ↓
extract_latest_retrieval_evidence
  ↓
validate_answer
  ↓
如果校验通过：
      publish final
  ↓
如果校验失败：
      rewrite_answer
      ↓
      validate_answer again
      ↓
      publish final
```

## 九、allowed_terms 解析逻辑

V0.8 会从最近一次 `retrieve_tcm_knowledge` 工具返回中解析 allowed_terms。

工具返回内容中包含：

```text
允许使用的专业术语：

- 胃胀
- 嗳气
- 反酸
- 烧心
- 恶心
- 便秘
- 腹泻
- 大便不成形
- 饮食不节
- 情志不畅
- 脾胃虚弱
- 气机不畅

回答约束：
- 回答中的中医专业术语应优先来自“允许使用的专业术语”。
```

V0.8 的解析逻辑是：

```text
1. 从 messages 后往前找最近一条 retrieve_tcm_knowledge tool message
2. 找到“允许使用的专业术语”这一行
3. 逐行收集以 “-” 开头的术语
4. 遇到“回答约束”停止
5. 返回 allowed_terms
```

这样可以保证使用的是最近一次检索结果，而不是旧轮次的检索结果。

---

## 十、evidence_text 解析逻辑

V0.8 同时会提取最近一次完整的 RAG 工具返回内容作为 `evidence_text`。

这个内容用于答案重写时提供证据上下文。

也就是说，重写模型看到的不只是 allowed_terms，还能看到：

```text
1. 检索模式
2. 原始检索问题
3. 改写后的检索问题
4. TopK 检索证据
5. 来源文件和 chunk_id
6. 检索分数
7. 允许使用的专业术语
8. 回答约束
```

这样可以让重写答案更贴合证据，而不是单纯删除术语。

## 十一、V0.8 final 事件升级

V0.8 的 `final` 事件新增三个字段：

```json
{
  "validation": {},
  "validation_before_rewrite": {},
  "rewritten": true
}
```

### 1. `validation`

表示最终输出答案的校验结果。

如果重写后通过，则为：

```json
{
  "passed": true,
  "reason": "ok",
  "answer_terms": [],
  "allowed_terms": [],
  "unsupported_terms": []
}
```

---

### 2. `validation_before_rewrite`

表示原始答案在重写前的校验结果。

如果原始答案存在无依据术语，则为：

```json
{
  "passed": false,
  "reason": "unsupported_terms_found",
  "unsupported_terms": [
    "肝胃不和",
    "寒湿困脾"
  ]
}
```

---

### 3. `rewritten`

表示本轮最终回答是否经过重写：

```json
"rewritten": true
```

如果没有触发重写，则为：

```json
"rewritten": false
```

## 十四、V0.8 的价值

V0.8 的价值主要体现在三个方面。

### 1. 降低术语幻觉

系统可以主动发现：

```text
模型回答中出现了 allowed_terms 之外的中医术语
```

比如：

```text
肝胃不和
寒湿困脾
食积内停
```

如果这些术语没有被检索证据支持，就会被识别为 unsupported_terms。

---

### 2. 提高知识可追溯性

V0.8 要求回答中的专业术语尽量来自：

```text
本轮 RAG 检索结果中的 allowed_terms
```

因此，最终回答更容易追溯到检索证据。

---

### 3. 增强系统可信度

相比单纯依靠 Prompt，V0.8 加入了工程层面的校验和重写机制。

这使系统从：

```text
让模型尽量别乱说
```

升级为：

```text
模型说完后，系统再检查一遍；不合格就重写
```

这对中医问答这类专业场景更安全。

---

## 十五、V0.8 对论文的表达价值

V0.8 可以在论文中表述为：

```text
知识依据约束与术语一致性校验机制
```

或者：

```text
面向中医问答的术语幻觉检测与答案重写机制
```

可写为：

```text
为降低大语言模型在中医问答过程中生成无依据专业术语的风险，本文在 Agentic RAG 流程中引入术语一致性校验机制。系统首先通过混合检索模块返回与用户问题相关的知识片段及允许使用的中医术语集合；随后，在答案生成后对回答中的症状、证候、病机及调护相关术语进行抽取，并与检索结果中的允许术语集合进行一致性比对。若发现回答中存在检索证据未支持的术语，系统将触发答案重写，使最终回答尽量基于已有知识证据生成，从而提升中医问答结果的可靠性与可追溯性。
```

---

## 十六、V0.8 当前不足

V0.8 虽然已经实现了术语校验与答案重写，但仍然存在以下不足：

```text
1. 术语抽取依赖人工维护词典，无法识别词表之外的新术语
2. 当前只校验术语是否在 allowed_terms 中，尚未判断术语使用语义是否正确
3. 重写仍然依赖 LLM，本身也可能产生新的表达偏差
4. 当前最多触发一次重写，没有多轮自我修正机制
5. 没有对处方剂量、绝对诊断表达、危险信号遗漏做系统校验
6. allowed_terms 目前从工具返回文本中解析，后续可以改为结构化状态存储
```

---

## 十七、后续优化方向

V0.8 后续可继续增强以下能力：

```text
1. 扩充中医术语词典
2. 将 allowed_terms 从文本解析改成结构化存储
3. 增加处方/剂量/诊断表达检测
4. 增加危险信号提醒检测
5. 增加重写后多轮校验机制
6. 引入中医 NER 模型或 LLM 术语抽取
7. 增加 answer validation 日志，方便后续实验统计
```

```text
多智能体协同推理
```

可以拆分为：

```text
问诊 Agent：判断信息是否充分，负责追问
检索 Agent：负责生成检索 query 并调用 RAG
辨证 Agent：基于证据进行谨慎辨析
调护 Agent：生成生活调护和就医提醒
Coordinator：汇总各智能体结果，进行一致性校验
```

V0.8 的术语校验机制可以作为 Coordinator 的最后一道安全校验层。

## 十九、V0.8 验收标准

V0.8 完成后，应满足：

```text
1. 新增 app/guardrails/term_checker.py
2. 新增 app/guardrails/answer_validator.py
3. 新增 app/guardrails/answer_rewriter.py
4. 能从最终回答中抽取 answer_terms
5. 能从 retrieve_tcm_knowledge 工具返回中解析 allowed_terms
6. 能计算 unsupported_terms
7. unsupported_terms 不为空时能触发答案重写
8. 重写后能再次执行术语校验
9. final 事件中返回 validation、validation_before_rewrite、rewritten
10. ask_clarification 追问流程不受影响
11. V0.7 hybrid search 功能不受影响
12. 重写答案中不应出现“我已重写”“重写后的答案如下”等内部过程性话语
```

## 二十、一句话总结

V0.8 的核心升级是：

```text
在混合检索增强之后，增加术语一致性校验与答案重写机制，使最终回答中的中医专业术语尽量受到检索证据约束。
```

该版本使系统从：

```text
检索增强生成
```

```text
证据约束生成
```

为后续多智能体协同推理和论文中的系统可信度设计奠定基础。
