# TCM-Flow V2.0 更新说明：多智能体 Workflow 与 LangGraph Checkpointer

日期：2026-06-25

## 1. 本版本目标

V2.0 的目标是把 `workflow_agent` 从普通顺序编排升级为可持久化、可追踪、可恢复的多智能体 Workflow。

本次更新围绕中医问诊与证据生成链路，将原先由代码手动串联的执行流程迁移到 LangGraph `StateGraph`：

```text
IntentAgent / InquiryAgent
  -> EvidenceAgent
  -> SyndromeAgent
  -> AnswerAgent
  -> SafetyAgent
  -> finalize
```

其中 LangGraph 负责任务节点、条件路由、内部消息状态和 checkpoint；`WorkflowAgent` 保留为运行时兼容适配层，继续对接现有 `run_agent`、SSE stream 和 `thread_store`。

## 2. 核心改动

### 2.1 新增 WorkflowState

新增 `app/agents/workflow_agent/state.py`，将多智能体运行态统一收敛到 `WorkflowState`：

```text
run_id
user_text
conversation
messages
intent
inquiry
evidence
syndrome
answer
safety
needs_clarification
final_text
agent_trace
```

其中：

1. `messages` 使用 LangGraph `add_messages` reducer，保留本轮 HumanMessage、AIMessage、ToolMessage 等执行消息；
2. `agent_trace` 使用追加式 reducer，记录每个 Agent 的关键决策；
3. `inquiry`、`evidence`、`syndrome`、`answer`、`safety` 等结构以可 JSON 序列化 dict 写入 graph state，恢复时再用 Pydantic 模型校验。

### 2.2 新增 LangGraph 工作流图

新增 `app/agents/workflow_agent/graph.py`，通过 `StateGraph(WorkflowState)` 构建固定多智能体图。

当前图中包含以下节点：

| 节点 | 职责 |
| --- | --- |
| `intent` | 判断用户输入的任务意图和后续路由 |
| `direct_response` | 对问候、离题等场景直接返回简短回复 |
| `inquiry` | 整理问诊信息，判断是否需要追问 |
| `clarification` | 生成标准 `ask_clarification` 工具调用消息 |
| `evidence` | 调用 `retrieve_tcm_knowledge` 获取证据 |
| `syndrome` | 仅基于允许术语和证据生成候选辨证方向 |
| `answer_draft` | 整合前序结果生成初稿 |
| `safety_initial` | 审查初稿是否存在诊断、处方、剂量或风险遗漏 |
| `answer_rewrite` | 按安全审查要求重写回答 |
| `safety_rewrite` | 复查重写后的回答 |
| `safe_fallback` | 多次不安全时生成保守兜底回答 |
| `finalize` | 写入最终 AIMessage 和 `final_text` |

### 2.3 条件路由改为图内决策

Workflow 的关键分支不再散落在手写顺序逻辑中，而是由 graph 路由函数统一控制：

```text
intent
  ├── direct_response
  ├── inquiry
  └── evidence

inquiry
  ├── clarification
  └── evidence

evidence
  ├── syndrome
  └── answer_draft

safety_initial
  ├── answer_rewrite
  └── finalize

safety_rewrite
  ├── safe_fallback
  └── finalize
```

这样每个 Agent 的职责和边界更清晰，也便于后续在图上增加新节点、调试中间状态或实现恢复执行。

## 3. Runtime 集成

### 3.1 TCMWorkflow 持有 CompiledStateGraph

`TCMWorkflow` 现在不再手动顺序调用各 Agent，而是在初始化时创建并持有编译后的 LangGraph：

```python
self.graph = build_workflow_graph(..., checkpointer=checkpointer)
```

运行时 streaming 通过：

```python
async for event in self.graph.astream(...):
    ...
```

`TCMWorkflow.run()` 也收敛为对 `graph.astream(..., stream_mode=["values"])` 的收集，不再绕回 `graph.ainvoke()` 执行整轮后一次性吐出结果。

### 3.2 WorkflowAgent 保留为 runtime adapter

`WorkflowAgent` 不是 LangGraph 官方 prebuilt agent，也不是 `compile()` 直接返回的 `CompiledStateGraph`。

它的定位是项目自定义 runtime adapter：

1. 从 `thread_store` 读取产品侧 conversation；
2. 提取本轮用户输入并调用 `TCMWorkflow.astream()`；
3. 透传 graph streaming 事件；
4. 把 `aget_state()` / `aupdate_state()` 直接委托给 compiled graph。

`WorkflowAgent` 不再直接写 `thread_store` 的本轮投影；最终 messages 和 conversation 由 `run_agent` runtime 统一落库。guardrail 中间结果和 agent trace 不再重复写入 thread metadata。

### 3.3 Checkpointer 与 thread_store 分工

V2.0 之后，两层存储边界如下：

| 层 | 职责 |
| --- | --- |
| LangGraph checkpointer | 保存内部执行状态、messages、agent_trace、节点中间结果 |
| `thread_store` | 保存产品侧投影，包括 conversation、最终回复、校验结果、可见历史和 run/thread 状态 |

也就是说，checkpointer 负责“Workflow 怎么执行”，`thread_store` 负责“产品怎么展示和恢复可见会话”。

## 4. 结构化输出兼容

### 4.1 OpenAI 与 DeepSeek 分流

`workflow_agent/llm.py` 新增结构化输出适配逻辑：

1. OpenAI 官方模型继续使用：

```python
with_structured_output(schema, method="json_schema", strict=True)
```

2. DeepSeek 或 DeepSeek-compatible base_url 使用：

```python
with_structured_output(method="json_mode")
```

这样可以避开 DeepSeek 当前不支持部分 `response_format` / `json_schema` 严格模式的问题。

### 4.2 JSON Mode Prompt 约束

`StructuredWorkflowComponent` 会在系统提示词中补充明确 JSON 输出要求，并附带当前 Pydantic schema 的 JSON schema。

这修复了以下模型侧报错：

```text
Prompt must contain the word 'json' in some form to use 'response_format' of type 'json_object'.
```

### 4.3 InquiryState 容错归一化

针对模型在 JSON Mode 下可能返回的非严格形态，增加了 `InquiryState` payload 归一化：

1. `known_facts` 如果被返回成字符串，会归入 `associated_symptoms`；
2. `missing_info` 如果是中文顿号或逗号分隔字符串，会拆成 list；
3. 顶层 `risk_flags` 会合并进 `known_facts.risk_flags`；
4. 多余字段会在进入 Pydantic 校验前过滤。

这避免模型输出可读但不完全贴合 schema 时直接抛出 `OutputParserException`。

## 5. 问诊澄清策略修复

本次也修复了一个实际任务流问题：用户询问“最近起床经常胃痛什么原因导致的”并多轮补充信息后，工作流仍可能持续停留在 `InquiryAgent -> clarification`，无法进入证据检索和回答生成。

修复后：

1. 对“什么原因 / 为什么 / 导致 / 怎么回事”等原因解释型咨询，只要已有明确主诉，不再无限追问；
2. 如果已有澄清上下文，且用户已经补充了有效事实，会继续进入 evidence；
3. 如存在风险信号，不再卡在澄清阶段，而是继续进入后续安全审查，由 `SafetyAgent` 保守处理；
4. 真正缺少主诉或信息严重不足的场景仍会进入 clarification。

## 6. 用户可见协议保持不变

V2.0 保持 V1.9 后收敛出的 SSE 事件边界：

```text
metadata
messages
updates
values
final
error
end
```

对前端来说，最终业务响应仍以 `final` 事件为准：

```python
{
    "assistant_message": "...",
    "pending_clarification": [...],
    "references": [...]
}
```

Workflow 内部可以变成多节点 graph，但产品层仍保持轻量、稳定的可见响应契约。

## 9. 结果

V2.0 之后，`workflow_agent` 的定位更加清晰：

1. 内部执行由 LangGraph `CompiledStateGraph` 管理；
2. 多智能体节点职责清楚，路由可观察；
3. checkpointer 保存 workflow 执行状态；
4. `thread_store` 保存产品侧可见投影；
5. `WorkflowAgent` 作为 runtime adapter 兼容现有 SSE 与 `run_agent`；
6. OpenAI 与 DeepSeek 的结构化输出差异被统一封装；
7. 原因类问诊不再容易陷入无限澄清。

这为后续 V2.1 的意图驱动路由、更多专业 Agent 扩展，以及更复杂的工作流恢复能力打下了基础。
