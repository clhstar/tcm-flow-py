# TCM-Flow V1.3 更新说明：Checkpointer 原生澄清流程

> 版本主题：将 V1.2 的 `pending_clarification + resume_prompt` 自定义恢复机制，重构为基于 LangGraph Checkpointer 和 Middleware 的原生多轮澄清流程。  
> 版本定位：减少 Runtime 自定义状态与启发式判断，让 Agent 通过结构化工具调用表达澄清意图，并由同一 Thread 的消息历史自然承接用户补充。

---

## 一、版本背景

V1.2 已经支持：

```text
信息不足
  ↓
触发 clarification
  ↓
Run 进入 waiting_clarification
  ↓
用户补充信息
  ↓
恢复执行
```

但 V1.2 为实现恢复流程，引入了较多 Runtime 自定义状态：

```text
pending_clarification
last_resume
is_resume
resume_prompt
final_fallback
required_fields 关键词推断
```

这套机制虽然能够运行，但存在以下问题：

```text
1. ThreadStore 和 LangGraph Checkpointer 同时维护对话状态；
2. 用户补充内容需要被重新包装成 resume_prompt；
3. Runtime 需要判断当前 Run 是否属于恢复执行；
4. final answer 中的普通问句可能被启发式规则误判为澄清；
5. 澄清字段依赖关键词和正则提取，不够稳定；
6. Worker 承担了过多 Agent 决策逻辑。
```

V1.3 的目标是删除这些重复状态和启发式流程，让 LangGraph Checkpointer 成为对话恢复的唯一历史来源。

---

## 二、核心变化

V1.3 将澄清流程调整为：

```text
Lead Agent 判断信息不足
  ↓
调用 ask_clarification(questions=[...])
  ↓
ClarificationMiddleware 拦截工具调用
  ↓
生成带 tool_call_id 的 ToolMessage
  ↓
Command(goto=END) 结束当前 Agent Run
  ↓
Runtime 发布 clarification 事件
  ↓
Run 状态变为 waiting_clarification
  ↓
用户在同一 Thread 中提交补充信息
  ↓
新 Run 只传入本轮用户消息
  ↓
LangGraph Checkpointer 自动合并历史消息
  ↓
Agent 继续判断、检索和回答
```

核心原则是：

```text
澄清状态由消息历史表达，
多轮恢复由 Checkpointer 完成，
Runtime 不再额外构造 resume_prompt。
```

## 三、结构化澄清工具

`ask_clarification` 的参数由单一字符串：

```python
ask_clarification(question: str)
```

调整为结构化问题数组：

```python
ask_clarification(questions: list[str])
```

调用示例：

```json
{
  "name": "ask_clarification",
  "args": {
    "questions": [
      "这种情况持续多久了？",
      "疼痛是否会向臀部或下肢放射？",
      "是否伴有下肢麻木或大小便异常？"
    ]
  }
}
```

约束规则：

```text
1. questions 必须是数组；
2. 每次包含 1 到 3 个问题；
3. 每个问题应当清晰且相互独立；
4. 不应重复询问用户已经回答过的信息。
```

部分模型可能把数组输出为 JSON 字符串，Middleware 会先尝试执行 JSON 解析。

---

## 四、新增 ClarificationMiddleware

新增文件：

```text
app/middlewares/clarification_middleware.py
```

主要职责：

```text
1. 拦截 ask_clarification 工具调用；
2. 校验 questions 参数；
3. 限制每次最多追问 3 个问题；
4. 将问题统一格式化为面向用户的追问文本；
5. 构造与原始 tool call 对应的 ToolMessage；
6. 使用 Command(goto=END) 结束当前 Agent 执行。
```

生成的 ToolMessage 会保留：

```text
id
name
tool_call_id
content
```

示例：

```json
{
  "id": "clarification:call-1",
  "name": "ask_clarification",
  "tool_call_id": "call-1",
  "content": "为了更准确地帮您分析，请先补充以下关键信息：\n1. ...\n2. ...\n3. ..."
}
```

保留 `tool_call_id` 可以维持 AI ToolCall 与 ToolMessage 之间的完整关联，避免破坏 LangGraph 的消息链。

---

## 五、删除 pending_clarification 状态

V1.3 不再向 Thread values 写入：

```text
pending_clarification
last_resume
```

Thread history 接口也不再返回这两个字段。

删除前：

```json
{
  "conversation": [],
  "messages": [],
  "pending_clarification": {},
  "last_resume": {}
}
```

删除后：

```json
{
  "conversation": [],
  "messages": [],
  "last_validation": {},
  "last_allowed_terms": [],
  "last_rewritten": false,
  "last_agent_trace": []
}
```

澄清上下文已经保存在 Checkpointer 的消息历史中，因此不再需要额外复制一份 pending 状态。

---

## 六、删除自定义 Resume 模块

V1.3 删除：

```text
app/runtime/resume.py
```

同时删除以下逻辑：

```text
has_pending_clarification
build_resume_input_data
build_resume_record
increment_resume_count
context.is_resume
resume_prompt
resume agent_step 事件
```

用户补充信息时，不再将输入包装为：

```text
用户正在回答上一轮澄清问题……
上一轮澄清问题……
需要补充的关键信息……
```

新 Run 只向 Agent 传入当前用户消息：

```python
{"messages": [{"role": "user", "content": user_text}]}
```

历史对话由相同 `thread_id` 对应的 LangGraph Checkpointer 自动提供。

---

## 七、移除 Final Fallback 启发式判断

V1.2 会扫描 final answer，通过关键词、问号数量和问诊字段判断它是否属于正式追问。

相关逻辑包括：

```text
FORMAL_CLARIFICATION_KEYWORDS
CLARIFICATION_STRUCTURE_PHRASES
FIELD_RULES
DEFAULT_FIELD_QUESTIONS
extract_questions_from_text
extract_required_fields
should_convert_final_to_clarification
build_clarification_from_final
```

V1.3 删除这些启发式规则。

新的判断方式只有一条：

```text
最后一条消息是 name=ask_clarification 的 ToolMessage
  ↓
当前 Run 进入 clarification 状态
```

这样可以避免普通 final answer 因为包含问句而被 Runtime 误判为澄清中断。

相应地，Agent Prompt 和工具选择必须保证：

```text
需要正式追问时必须调用 ask_clarification，
不能只在 final answer 中书写一组问题。
```

## 八、Worker 职责简化

`app/runtime/runs/worker.py` 删除了：

```text
1. pending_clarification 检测；
2. resume input 构造；
3. resume record 保存；
4. final_fallback 转换；
5. required_fields 提取；
6. final 后清空 pending 状态。
```

Worker 当前只负责：

```text
1. 创建 Agent；
2. 传入本轮用户消息；
3. 执行 agent.astream；
4. 推送 values 和 agent_step；
5. 检测 ask_clarification ToolMessage；
6. 保存 messages 与 conversation；
7. 更新 Run / Thread 状态；
8. 执行 Guardrails 并返回 final。
```

这使 Runtime 更接近：

```text
Agent 决策
Middleware 控制工具行为
Checkpointer 保存对话状态
Worker 管理生命周期和事件
```

## 九、消息序列化增强

`message_to_dict()` 新增保留：

```text
message.id
tool_call_id
```

修改前，序列化结果主要包含：

```json
{
  "type": "tool",
  "content": "...",
  "name": "ask_clarification"
}
```

修改后：

```json
{
  "type": "tool",
  "content": "...",
  "name": "ask_clarification",
  "id": "clarification:call-1",
  "tool_call_id": "call-1"
}
```

这项调整便于：

```text
1. 调试 ToolCall 与 ToolMessage 的关联；
2. 验证 Checkpointer 中的工具调用链；
3. 前端或日志系统追踪具体工具调用；
4. 后续按 message id 更新或替换消息。
```

## 十、接口与兼容性变化

### 1. ask_clarification 参数变化

旧格式：

```json
{
  "question": "请补充持续时间。"
}
```

新格式：

```json
{
  "questions": [
    "请补充持续时间。"
  ]
}
```

这是一次工具 Schema 变更。所有 Prompt、测试、手动 ToolCall 和模型 Few-shot 示例都应同步使用 `questions`。

### 2. Thread history 字段变化

删除：

```text
pending_clarification
last_resume
```

前端不应再依赖这些字段判断等待状态，应使用：

```text
thread.status == "waiting"
run.status == "waiting_clarification"
SSE event == "clarification"
```

### 3. 不再发布 Resume 专用事件

用户补充信息会作为同一 Thread 中的新 Run 正常执行，不再发布：

```text
agent_step.type = resume
```

## 十一、新增测试

新增：

```text
tests/__init__.py
tests/test_clarification_flow.py
```

测试目标包括：

```text
1. 澄清问题不能为空；
2. 每轮最多输出 3 个问题；
3. ToolMessage 序列化保留 id；
4. ToolMessage 序列化保留 tool_call_id；
5. 首次 Run 可以进入 waiting_clarification；
6. clarification SSE 事件可以正常发送；
7. 用户在同一 Thread 中补充信息后可继续执行；
8. Checkpointer 能保留 human / ai / tool 消息链；
9. 第二次 Run 可以正常返回 final。
```

预期消息链：

```text
HumanMessage：用户首次提问
  ↓
AIMessage：调用 ask_clarification
  ↓
ToolMessage：结构化澄清问题
  ↓
HumanMessage：用户补充信息
  ↓
AIMessage：继续分析或最终回答
```

## 十二、V1.2 与 V1.3 对比

| 对比项 | V1.2 | V1.3 |
|---|---|---|
| 澄清输入 | `question: str` | `questions: list[str]` |
| 中断位置 | Worker 主动构造 payload | Middleware 拦截工具调用 |
| 澄清状态 | `pending_clarification` | Checkpointer 消息历史 |
| 恢复方式 | `resume_prompt` 包装输入 | 同一 Thread 新 Run |
| 恢复标记 | `is_resume` | 不需要 |
| 恢复记录 | `last_resume` | 不需要 |
| Final 追问识别 | 关键词与正则兜底 | 必须调用工具 |
| 缺失字段 | Runtime 关键词推断 | 由 Agent 生成结构化问题 |
| 历史来源 | ThreadStore + Checkpointer | Checkpointer 为主 |
| Worker 职责 | 决策与生命周期混合 | 主要负责生命周期 |

## 十三、架构意义

V1.3 的核心并不是简单删除字段，而是明确状态所有权：

```text
LangGraph Checkpointer：
负责 Agent 对话历史和工具调用链

ThreadStore：
负责面向 API 的 Thread 状态和可见 conversation

RunManager：
负责单次 Run 的生命周期

ClarificationMiddleware：
负责 ask_clarification 的校验、格式化和中断

Worker：
负责执行、事件推送和状态同步
```

这种职责划分减少了：

```text
1. 重复状态；
2. 自定义恢复协议；
3. Runtime 对自然语言的启发式判断；
4. Worker 中与 Agent 决策相关的业务代码。
```

---

## 十四、已知限制

V1.3 当前仍有以下边界：

```text
1. Agent 必须主动调用 ask_clarification；
2. final answer 中的正式追问不再自动转换为 clarification；
3. 前端无法再通过 pending_clarification 获取 required_fields；
4. questions 的完整性仍由模型负责；
5. 当前 Middleware 只校验问题数量和基本格式；
6. InMemorySaver 在进程重启后不会保留历史；
7. 多实例部署时仍需要持久化 Checkpointer。
```

因此，Prompt 中必须继续强调：

```text
信息不足时调用 ask_clarification，
调用后停止当前回答，
用户补充后再继续检索和分析。
```

---

## 十五、验收标准

V1.3 应满足：

```text
1. ask_clarification 使用 questions 数组；
2. questions 数量限制为 1 到 3 个；
3. Agent 注册 ClarificationMiddleware；
4. 工具调用后生成带 tool_call_id 的 ToolMessage；
5. 当前 Run 状态变为 waiting_clarification；
6. Thread 状态变为 waiting；
7. SSE 返回 clarification 事件；
8. 用户补充信息时不再构造 resume_prompt；
9. 同一 thread_id 的 Checkpointer 历史能够被继续使用；
10. 第二次 Run 能够继续生成 final；
11. history 接口不再返回 pending_clarification；
12. history 接口不再返回 last_resume；
13. Runtime 不再依赖 final_fallback；
14. messages 序列化保留 id 和 tool_call_id。
```

---

## 十六、一句话总结

V1.3 的核心升级是：

```text
将 V1.2 的自定义 pending/resume 澄清流程，
重构为“结构化 ask_clarification + Middleware 中断 +
LangGraph Checkpointer 原生多轮历史”的澄清机制。
```

这使 TCM-Flow 的澄清流程从 Runtime 自定义状态机，进一步靠近由 Agent、Middleware 和 Checkpointer 共同驱动的 DeerFlow-like 架构。
