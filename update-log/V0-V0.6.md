# V0.0：DeerFlow 架构理解与 Mini 版规划

## 1. 版本目标

V0.0 主要不是写代码，而是理解 DeerFlow 的整体架构，并规划 Mini-DeerFlow 的简化实现路线。

DeerFlow 的主链路可以概括为：

```text
chat.sh
  ↓
/api/langgraph/threads/{thread_id}/runs/stream
  ↓
Gateway / FastAPI Router
  ↓
services.py
  ↓
runtime worker.py
  ↓
lead_agent
  ↓
tools / middleware / subagents
  ↓
LangGraph 执行
  ↓
SSE 返回
```

Mini 版本的目标是先保留核心主干：

```text
FastAPI 接口
  ↓
Thread / Run 管理
  ↓
SSE 流式响应
  ↓
Lead Agent
  ↓
Tools
  ↓
LangGraph 执行
```

## 2. DeerFlow 关键模块理解

| DeerFlow 模块 | 作用 | Mini 版对应 |
|---|---|---|
| `gateway/app.py` | FastAPI 应用入口与路由注册 | `app/main.py` |
| `routers/thread_runs.py` | Run 流式执行入口 | `/api/threads/{thread_id}/runs/stream` |
| `services.py` | 创建 Run，准备执行上下文 | `app/services.py` |
| `worker.py` | 真正执行 Agent，并推送 SSE | `app/worker.py` |
| `lead_agent/agent.py` | 组装模型、工具、Prompt | `app/agents/lead_agent.py` |
| `lead_agent/prompt.py` | 定义 Agent 行为规则 | `app/agents/prompt.py` |
| `tools/` | 工具注册与工具函数 | `app/tools/` |
| Checkpointer | 保存 Thread 内状态 | LangGraph `InMemorySaver` |
| StreamBridge | 连接后台任务和 SSE | `app/stream.py` |

# V0.2：Final 事件与 Thread 查看接口

V0.2 增加：

```text
event: final
GET /api/threads/{thread_id}

```

## 2. 新增 final 事件

在 `worker.py` 中增加：

```python
def extract_final_ai_text(messages: list[dict]) -> str:
    for msg in reversed(messages):
        if msg.get("type") == "ai" and msg.get("content"):
            if not msg.get("tool_calls"):
                return msg["content"]
    return ""
```

Run 执行完成后推送：

```text

data: {
  "content": "...",
  "messages": [...]
}
```

# V0.3：ask_clarification 澄清中断机制

## 1. 版本目标

V0.3 的目标是实现真正的问诊追问机制：

```text
信息不足
  ↓
Agent 调用 ask_clarification
  ↓
工具执行完成
  ↓
worker 检测到 ask_clarification tool message
  ↓
返回 event: clarification
  ↓
当前 Run 中断
  ↓
Thread 状态变为 waiting
```

## 2. 为什么需要中断机制？

中医问诊中经常需要追问：

```text
持续多久？
是否伴随嗳气、反酸？
大便情况如何？
是否有危险信号？
```

如果信息不足时模型直接给结论，容易产生幻觉或不严谨回答。

因此需要：

```text
先追问，再分析
```

## 3. ask_clarification 工具升级

工具定义：

```python
@tool("ask_clarification", return_direct=True)
def ask_clarification(question: str) -> str:
    """
    当用户信息不足、症状描述不清楚、需要补充病史或确认风险时调用。
    调用该工具后，本轮对话应该暂停，等待用户回答。
    """
    return question
```

## 4. Worker 中断检测

关键逻辑：

```python
def extract_latest_clarification_question(messages: list[dict]) -> str:
    if not messages:
        return ""

    latest = messages[-1]

    if latest.get("type") == "tool" and latest.get("name") == "ask_clarification":
        content = latest.get("content", "")
        if isinstance(content, str) and content.strip():
            return content.strip()

    return ""
```

注意：不能在 AI 刚产生 `tool_calls` 时中断，必须等工具执行后出现 tool message 再中断。

原因是 OpenAI 工具调用格式要求：

```text
assistant message with tool_calls
  ↓
must be followed by tool messages
```

如果提前中断，下一轮会报：

```text
An assistant message with 'tool_calls' must be followed by tool messages
```

## 5. V0.3 SSE 输出

正常输出：

```text
event: metadata
data: {"run_id":"...","thread_id":"..."}

event: values
data: {"messages":[...]}

event: clarification
data: {
  "question":"请问您这个胃胀持续多久？是否伴随嗳气、反酸、大便异常？",
  "thread_id":"...",
  "run_id":"..."
}

event: end
data: {"status":"done"}
```

## 6. V0.3 提升点

| 提升点 | 说明 |
|---|---|
| 澄清追问 | 信息不足时先追问 |
| Run 中断 | 当前执行停止，等待用户补充 |
| Thread waiting 状态 | 前端可识别正在等待用户 |
| 医疗安全性提升 | 避免信息不足时直接给结论 |

## 7. V0.3 的不足

```text
1. 多轮上下文仍需要完善
2. 如果自己手动拼接历史，容易和 LangGraph Checkpointer 冲突
3. 知识库仍然不是真实 RAG
```

# V0.4：多轮上下文管理

## 1. 版本目标

V0.4 解决多轮问诊问题。

目标：

```text
第一轮：
用户：我胃胀，饭后明显
系统：请问持续多久？是否嗳气、反酸、大便异常？

第二轮：
用户：持续两周，有嗳气，大便不成形
系统：结合上一轮继续分析
```

## 2. 关键问题

早期实现中同时使用了两套历史机制：

```text
1. LangGraph Checkpointer 自动保存 messages
2. thread_store.conversation 手动拼接历史
```

这会导致第二轮上下文重复：

```text
LangGraph 历史
+
手动拼接历史
+
当前用户输入
```

最终出现重复追问或消息重复。

## 3. 最终方案

V0.4 确定采用更接近 DeerFlow 的方式：

```text
LangGraph Checkpointer：负责真实模型上下文
thread_store.conversation：只负责前端展示
```

也就是说：

```text
不要手动把 conversation 拼回 Agent input
每次只传当前用户输入
通过 config.configurable.thread_id 让 LangGraph 自动接续历史
```

## 4. Agent Checkpointer

`lead_agent.py` 中保留：

```python
from langgraph.checkpoint.memory import InMemorySaver

_checkpointer = InMemorySaver()

agent = create_agent(
    model=model,
    tools=tools,
    system_prompt=SYSTEM_PROMPT,
    checkpointer=_checkpointer,
)
```

## 5. Worker 中保留 thread_id config

```python
config = {
    "configurable": {
        "thread_id": thread_id,
    },
    "recursion_limit": context.get("recursion_limit", 50),
}
```

## 6. 当前输入只传本轮消息

```python
messages = normalize_messages(input_data)
```

不再使用：

```python
messages = build_agent_messages(thread_values, input_data)
```

## 7. conversation 的定位

`conversation` 仍然保存，但只用于：

```text
1. 前端聊天历史展示
2. GET /api/threads/{thread_id}/history
3. 调试用户可见对话
```

不作为模型上下文来源。

## 8. V0.4 提升点

| 提升点 | 说明 |
|---|---|
| 多轮上下文 | 同一个 thread_id 下自动接续历史 |
| 更接近 DeerFlow | 使用 LangGraph Checkpointer 管理状态 |
| 避免重复历史 | conversation 不再拼回模型输入 |
| 前端展示清晰 | conversation 只保存用户可见内容 |

## 9. V0.4 的不足

```text
1. Checkpointer 目前是内存版，服务重启后历史丢失
2. 还没有持久化数据库
3. 知识检索仍然是写死 dict 或基础工具
```

# V0.6：RAG 检索质量优化

## 1. 版本目标

V0.6 的目标是把基础 RAG 升级成“检索更准、结果更可控、过程可追踪”的 RAG 优化版本。

核心能力：

```text
1. Chunk Metadata 增强
2. Query Rewrite 查询改写
3. 简单 Rerank 规则重排
4. Allowed Terms 专业术语约束
5. Retrieval Log 检索日志
```

## 2. V0.6 检索流程

```text
用户问题
  ↓
大模型自动生成工具 query
  ↓
retrieve_tcm_knowledge
  ↓
rewrite_tcm_query 规则扩展
  ↓
Chroma 向量召回 candidate_k
  ↓
规则 Rerank
  ↓
提取 allowed_terms
  ↓
返回 evidence + source + allowed_terms
  ↓
Agent 基于证据回答或追问
```

## 3. Query Rewrite

新增：

```text
app/rag/terms.py
```

功能：

```text
1. 维护 TCM_TERMS 中医术语表
2. 维护 QUERY_EXPANSIONS 查询扩展词
3. extract_terms 抽取术语
4. rewrite_tcm_query 改写检索 query
5. detect_topic 识别主题
```

示例：

```text
原始检索问题：
胃胀 饭后加重 中医 症状 病机 调护

改写后检索问题：
胃胀 饭后加重 中医 症状 病机 调护 胃脘胀满 嗳气 反酸 烧心 恶心 大便不成形 饮食不节 情志不畅 脾胃虚弱 气机不畅 问诊要点 日常调护
```

## 4. Chunk Metadata 增强

`documents.py` 从普通切分升级为 Markdown 标题解析：

```text
# 胃胀
  → topic = 胃胀

## 胃胀问诊要点
  → section = 胃胀问诊要点
```

每个 chunk metadata 包括：

```json
{
  "source": "data/raw/tcm_knowledge.md",
  "filename": "tcm_knowledge.md",
  "topic": "胃胀",
  "section": "胃胀问诊要点",
  "chunk_id": 1
}
```

## 5. 简单 Rerank

V0.6 的重排策略：

```text
rerank_score =
  keyword_hits * 2
  + topic_bonus
  + section_bonus
  - distance_score
```

其中：

```text
keyword_hits：query 术语和 chunk 术语的交集数量
topic_bonus：主题匹配加分
section_bonus：问诊 / 调护 / 危险信号章节加分
distance_score：向量距离，越小越好
```

## 6. Allowed Terms 专业术语约束

系统从最终检索结果中抽取可用术语：

```text
允许使用的专业术语：
- 胃胀
- 嗳气
- 反酸
- 烧心
- 恶心
- 饮食不节
- 情志不畅
- 脾胃虚弱
- 气机不畅
```

回答约束：

```text
回答中的中医专业术语应优先来自“允许使用的专业术语”。
如果检索结果中没有出现某个证候、病机或治法术语，不要主动引入。
检索结果只能作为健康咨询依据，不能作为诊断结论。
```


记录内容：

## 8. V0.6 示例效果

用户：

```text
我最近胃胀，吃完饭更明显，怎么办？
```

Agent 自动生成工具 query：

```text
胃胀 饭后加重 中医 症状 病机 调护
```

RAG 改写后：

```text
胃胀 饭后加重 中医 症状 病机 调护 胃脘胀满 嗳气 反酸 烧心 恶心 大便不成形 饮食不节 情志不畅 脾胃虚弱 气机不畅 问诊要点 日常调护
```

检索排序：

```text
[1] 胃胀问诊要点
[2] 胃胀概述
[3] 胃胀日常调护
```

然后 Agent 判断信息不足，调用：

```text
ask_clarification
```

返回：

```text
请问您这个胃胀的情况持续多久了？除了胃胀，有没有伴随嗳气、反酸、烧心、恶心，或者大便方面的变化？
```

## 9. V0.6 提升点

| 提升点 | 说明 |
|---|---|
| Query Rewrite | 将口语问题扩展为专业检索 query |
| Metadata | 每个 chunk 有 topic / section / source |
| Rerank | 向量召回后按规则重新排序 |
| Allowed Terms | 限制专业术语来源，降低幻觉 |
| Retrieval Log | 检索过程可追踪、可调试、可用于论文分析 |
| Agentic RAG 强化 | Agent 决定是否检索、检索什么、是否追问 |

## 10. V0.6 的不足

```text
1. 仍然只有向量检索，没有 BM25
2. Rerank 还是规则版，不是模型重排
3. Allowed Terms 只是提示约束，还没有强制术语校验
4. 知识库规模仍然较小
5. Checkpointer 仍然是内存版本，未持久化
```

# 后续版本建议

## V0.7：Hybrid Search 混合检索

目标：

```text
向量检索 + BM25 关键词检索 + 结果融合 + 重排
```

架构：

```text
Query Rewrite
  ↓
并行检索
  ├─ Dense Vector Search
  └─ BM25 Keyword Search
  ↓
Result Fusion
  ↓
Rerank
  ↓
Evidence + Allowed Terms
```

价值：

```text
1. 向量检索解决语义相近
2. BM25 解决中医术语精确匹配
3. 结果融合提升召回率与稳定性
4. 更贴合“混合检索增强”的论文题目
```

## V0.8：术语校验与答案重写

目标：

```text
生成后检查回答中的专业术语是否来自知识库证据
```

流程：

```text
Agent 生成回答
  ↓
抽取回答中的中医术语
  ↓
与 allowed_terms 比对
  ↓
发现 unsupported terms
  ↓
触发重写或删除无依据术语
```

价值：

```text
降低中医专业术语幻觉，提高知识可追溯性。
```
