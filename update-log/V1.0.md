# TCM-Flow V1.0 更新细节：Dynamic SubAgentExecutor 动态子任务委派机制

## 一、版本主题

V1.0 的主题是：

```text
Dynamic SubAgentExecutor 动态子任务委派机制
```

该版本是在 V0.9 DeerFlow-like Runtime 架构重构的基础上，进一步实现类似 DeerFlow 的 `task` 工具与动态子智能体执行机制。

V0.9 主要解决：

```text
系统架构如何对齐 DeerFlow：
Gateway → Runtime → Worker → Agent Registry → Lead Agent → Tools → Middleware
```

V1.0 主要解决：

```text
Lead Agent 如何动态拆解复杂任务，并通过 task 工具委派给临时子智能体执行。
```

该版本使系统从：

```text
Lead Agent + 工具调用
```

升级为：

```text
Lead Agent + 动态任务委派 + SubAgentExecutor + Agent Trace
```

---

## 二、V1.0 的设计思想

V1.0 最重要的设计原则是：**不写死固定业务子智能体，而是学习 DeerFlow 的动态任务委派思想。**

早期可以设计为：

```text
问诊 Agent
  ↓
辨证 Agent
  ↓
调护 Agent
  ↓
Coordinator
```

但这种方式本质上更像固定工作流，代码提前决定了任务顺序和角色分工。

而 DeerFlow-like 的设计更强调：

```text
Lead Agent
  ↓
根据当前任务动态判断下一步动作
  ↓
如果信息不足：调用 ask_clarification
  ↓
如果需要知识依据：调用 retrieve_tcm_knowledge
  ↓
如果任务复杂：调用 task 工具委派临时子任务
  ↓
收到子任务结果后继续综合或继续调用工具
```

因此，V1.0 不再预设固定的：

```text
tcm_consultation_agent
tcm_reasoning_agent
tcm_care_agent
```

而是引入：

```text
DynamicSubAgent
```

它不是固定业务角色，而是由 Lead Agent 通过 `task(description, expected_output)` 临时创建出来的子任务执行者。

---

## 三、V1.0 的核心目标

V1.0 的核心目标包括：

```text
1. 实现 task 工具的真实任务委派能力
2. 实现 DynamicSubAgentExecutor
3. 实现 DynamicSubAgent 临时子智能体
4. 由 Lead Agent 动态生成 task description
5. 子智能体根据 description 执行局部任务
6. 子智能体结果以 subagent_result 返回给 Lead Agent
7. Worker 从 task 工具结果中提取 agent_trace
8. final 事件返回 agent_trace，用于前端过程可视化和论文可解释性设计
```

最终流程如下：

```text
用户输入
  ↓
Lead Agent
  ↓
retrieve_tcm_knowledge / ask_clarification / task
  ↓
task(description, expected_output)
  ↓
SubAgentExecutor
  ↓
DynamicSubAgent
  ↓
subagent_result
  ↓
Lead Agent 综合
  ↓
Guardrails 术语校验
  ↓
final + agent_trace
```

## 五、核心模块一：`task` 工具

### 1. V0.9 中的 task 工具

V0.9 中的 `task_tool.py` 只是架构占位：

```python
@tool("task")
def task(description: str, agent_name: str = "tcm_subagent") -> str:
    return f"子任务已接收：agent={agent_name}, description={description}"
```

这个版本只能证明 Lead Agent 可以看到 `task` 工具，但没有真正执行子智能体任务。

---

### 2. V1.0 中的 task 工具

V1.0 将 task 工具升级为真实委派工具：

```python
@tool("task")
async def task(
    description: str,
    expected_output: str = "",
) -> str:
    executor = SubAgentExecutor()

    return await executor.run(
        description=description,
        expected_output=expected_output,
    )
```

该工具不再接收固定的 `agent_name`，而是只接收：

```text
description：子任务描述
expected_output：期望输出格式
```

这样 Lead Agent 可以根据当前任务动态描述子任务。

---

### 3. task 工具的职责

`task` 工具的职责是：

```text
1. 接收 Lead Agent 动态生成的子任务描述
2. 调用 SubAgentExecutor
3. 启动 DynamicSubAgent 执行任务
4. 返回结构化 subagent_result
5. 将子任务结果交还给 Lead Agent 综合
```

它不负责最终回答用户，也不直接替代 Lead Agent。

---

## 六、核心模块二：DynamicSubAgent

### 1. 设计定位

`DynamicSubAgent` 是 V1.0 的核心。

它不是固定角色，而是一个临时任务执行者。

它的任务不是预先写死的，而是由 Lead Agent 动态传入：

```text
请整理用户关于胃胀的已知问诊信息和缺失信息。
```

或者：

```text
请基于检索证据，总结胃胀、嗳气、恶心可能相关的因素，不能直接诊断。
```

或者：

```text
请根据检索证据整理适合用户的日常调护建议和就医提醒。
```

也就是说，子智能体的角色由 `description` 临时定义。

---

### 2. DynamicSubAgent 的系统提示词

`DynamicSubAgent` 的系统提示词强调：

```text
你是一个临时子智能体，由 Lead Agent 委派来完成一个明确的局部任务。

你的职责：
1. 只完成任务描述中要求的内容。
2. 不要扩展任务范围。
3. 不要直接下诊断。
4. 不要开处方。
5. 不要给出药物剂量。
6. 如果任务涉及中医知识，必须保持谨慎表达。
7. 如果任务描述中包含检索证据，应优先基于检索证据回答。
8. 如果证据不足，应明确说明“目前依据有限”。
9. 输出应清晰、简洁、结构化，方便 Lead Agent 后续综合。
```

这个设计保证了子智能体只做局部任务，不直接向用户输出最终结论。

---

## 七、核心模块三：SubAgentExecutor

### 1. 功能定位

`SubAgentExecutor` 是子智能体执行器。

它的职责是：

```text
1. 创建 DynamicSubAgent
2. 将 description 和 expected_output 传给子智能体
3. 等待子智能体执行完成
4. 将结果封装为 JSON 字符串
5. 返回给 task 工具
```

---

### 2. 返回格式

V1.0 中子智能体返回统一结构：

```json
{
  "type": "subagent_result",
  "agent_name": "dynamic_subagent",
  "task_description": "...",
  "expected_output": "...",
  "content": "...",
  "metadata": {
    "subagent_type": "dynamic"
  }
}
```

该结构有两个作用：

```text
1. 供 Lead Agent 阅读并综合最终回答
2. 供 Worker 提取 agent_trace
```

## 八、Lead Agent 的动态任务委派规则

V1.0 在 `app/agents/lead_agent/prompt.py` 中新增动态任务委派规则：

```text
动态子任务委派规则：
- 当 subagent_enabled=true，且用户问题较复杂时，你可以调用 task 工具委派一个临时子任务。
- task 工具不会调用固定业务 Agent，而是根据你提供的 description 创建动态子智能体。
- description 必须写清楚：
  1. 子任务目标；
  2. 已知上下文；
  3. 需要遵守的边界；
  4. 是否需要基于检索证据；
  5. 不允许直接诊断、不允许开处方。
- expected_output 用于说明你希望子智能体返回什么格式。
- 子智能体结果只是辅助分析，不能直接作为最终回答。
- 收到 task 工具结果后，你需要结合用户上下文、检索证据和工具结果，综合生成最终回答。
- 如果信息不足，应优先调用 ask_clarification，而不是强行委派子任务。
- 如果需要中医知识依据，应优先调用 retrieve_tcm_knowledge，再将检索证据作为上下文传入 task。
- 最终回答仍必须遵守 allowed_terms 和术语校验规则。
```

这样 Lead Agent 就具备了动态拆解任务的能力。

## 九、Tools 注册机制变化

V1.0 保持 V0.9 的工具注册中心设计：

```python
def get_available_tools(context: dict[str, Any] | None = None):
    context = context or {}

    tools = [
        ask_clarification,
        retrieve_tcm_knowledge,
        present_files,
    ]

    subagent_enabled = context.get("subagent_enabled", False)

    if subagent_enabled:
        tools.append(task)

    return tools
```

也就是说：

```text
subagent_enabled=false：
  Lead Agent 只能使用 ask_clarification、retrieve_tcm_knowledge、present_files

subagent_enabled=true：
  Lead Agent 额外获得 task 工具，可以委派动态子任务
```

这种方式对齐 DeerFlow 的工具动态注册思想。

## 十、Agent Trace 推理过程记录

V1.0 新增：

```text
app/middlewares/trace_middleware.py
```

用于从 `messages` 中解析 `task` 工具返回的 `subagent_result`，形成 `agent_trace`。

解析逻辑：

```text
1. 遍历 messages
2. 找到 type=tool 且 name=task 的消息
3. 将 content 解析为 JSON
4. 判断 type 是否为 subagent_result
5. 提取 agent_name、task_description、expected_output、content
6. 生成 agent_trace
```

最终 trace 示例：

```json
[
  {
    "agent": "dynamic_subagent",
    "action": "complete",
    "task_description": "请整理用户关于胃胀的已知问诊信息和缺失信息...",
    "expected_output": "请用要点输出：已知信息、缺失信息、风险信号。",
    "summary": "已知信息：用户主诉胃胀，饭后加重，伴嗳气和恶心..."
  }
]
```

## 十一、Worker 改造

V1.0 在 `app/runtime/runs/worker.py` 中增加 agent trace 提取。

在 final 阶段：

```python
agent_trace = extract_agent_trace_from_messages(final_messages)
```

保存到 thread values：

```python
await thread_store.update_values(
    thread_id,
    {
        "messages": final_messages,
        "conversation": conversation,
        "pending_clarification": None,
        "last_validation": validation,
        "last_allowed_terms": allowed_terms,
        "last_rewritten": rewritten,
        "last_agent_trace": agent_trace,
    },
)
```

并在 final 事件中返回：

```json
{
  "content": "...",
  "validation": {},
  "rewritten": false,
  "allowed_terms": [],
  "agent_trace": []
}
```

这样前端可以展示类似：

```text
正在检索中医知识……
正在委派子任务……
子智能体正在整理问诊信息……
正在进行答案校验……
```

这直接对应论文中的“推理过程可视化”。

## 十五、当前存在的问题

### 1. 信息不足时没有严格调用 ask_clarification

测试中，最终回答里出现了：

```text
为了更准确地帮您分析，请您先告诉我……
```

但 Lead Agent 没有调用 `ask_clarification` 工具，而是把追问内容写进了 final answer。

这会导致工程状态没有进入：

```text
thread status = waiting
event: clarification
```

因此需要进一步强化 prompt：

```text
如果你已经判断需要用户补充信息，必须调用 ask_clarification，不要把多个追问问题直接写在 final answer 中。
```

---

### 2. 部分中医术语可能漏检

测试回答中出现了：

```text
胃气壅滞
胃气当降不降
反而上逆
```

这些表达如果没有加入 `EXTRA_TCM_TERMS`，V0.8 术语校验可能不会识别出来。

建议扩充：

```python
EXTRA_TCM_TERMS = [
    "胃气壅滞",
    "胃气不降",
    "胃气当降不降",
    "胃失和降",
    "中焦气滞",
    "脾失健运",
]
```

这样可以进一步降低术语漏检风险。

---

### 3. 单个 task 承担了多个子任务

测试中 task description 同时包含：

```text
1. 整理问诊信息
2. 分析可能相关因素
```

这虽然可以运行，但 trace 不够细。

后续可以让 Lead Agent 更倾向于拆成多个 task：

```text
task 1：整理已知问诊信息和缺失信息
task 2：基于检索证据总结可能相关因素
task 3：整理日常调护建议和就医提醒
```

这样前端可视化和论文展示效果会更好。

## 十八、V1.0 与固定多智能体方案的区别

固定多智能体方案是：

```text
问诊 Agent → 辨证 Agent → 调护 Agent → Coordinator
```

V1.0 当前方案是：

```text
Lead Agent → task(description) → DynamicSubAgent → Lead Agent 综合
```

区别在于：

| 对比项          | 固定多智能体 | V1.0 动态任务委派     |
| ------------ | ------ | --------------- |
| 任务顺序         | 代码写死   | Lead Agent 动态决定 |
| 子智能体角色       | 固定     | 临时生成            |
| 灵活性          | 较低     | 较高              |
| DeerFlow 相似度 | 一般     | 更高              |
| 适合学习架构       | 一般     | 更适合             |

```text
1. 强化 Prompt，使信息不足时优先调用 ask_clarification
2. 让 Lead Agent 将复杂任务拆成多个 task，而不是一个 task 包含多个任务
3. 将 agent_trace 做成更丰富的事件流，而不是只在 final 中返回
4. 为 task 工具增加 max_subtasks 或 task_type 等可选字段
5. 为 SubAgentExecutor 增加并发执行能力
6. 为 DynamicSubAgent 增加 retrieval evidence 结构化输入
7. 统计子任务调用次数、重写次数、校验失败次数，用于论文实验分析
8. 后续可沉淀常用任务模板，但不要破坏动态委派主线
```

## 二十三、一句话总结

V1.0 的核心升级是：

```text
将 V0.9 中占位的 task 工具升级为真实的动态子任务委派机制，使 Lead Agent 能够根据当前问题动态创建子任务，并通过 SubAgentExecutor 调用 DynamicSubAgent 完成局部分析。
```

该版本使系统从：

```text
单一 Lead Agent 工具调用
```

进一步升级为：

```text
Lead Agent 动态规划 + task 工具委派 + 临时子智能体执行 + agent_trace 可追踪
```

这标志着系统的多智能体协同推理能力开始真正落地，并且比固定角色流水线更接近 DeerFlow 的架构思想。
