## 1. 研究目的

Formal-400 已经完成受控古籍检索实验，但它的题目由本项目围绕本地古籍语料构建。为了降低“结论只在自建问题集上成立”的质疑，本阶段引入第三方人工标注的天池中医文献问答数据，验证现有检索与 Parent-Child 方法是否能够迁移到不同来源、不同文体的中医语料。

本阶段同时解决两个不同问题：

1. **外部泛化：** B1-B4 与 P 在第三方人工 QA 上是否仍保持可复现的检索表现？
2. **Parent-Child 贡献：** 通用 Parent recovery 和中医结构识别分别对检索、上下文完整性及回答忠实度产生什么影响？

本阶段不重新解释或覆盖 Formal-400 结果。Formal-400 仍是古籍结构感知方法的受控主实验；天池实验是独立的外部效度验证。

## 3. 与论文研究问题的关系

本阶段主要补充以下研究问题：

- **RQ2：** 在外部中医 QA 上，Hybrid 是否仍优于单路检索？
- **RQ3：** 在外部中医 QA 上，Reranker 是否仍能改善前列证据排序？
- **RQ4：** 在具有真实结构信号的外部文段中，结构感知 Child 是否优于固定切分和通用 Parent-Child？
- **RQ5：** 检索 Child 后恢复 Parent，是否能改善证据完整性和回答忠实度？

本阶段不能单独回答：

- **RQ1：** 是否优于不使用外部知识的 LLM，需要独立的回答层 B0 对照；
- **RQ6：** 天池数据没有可靠的无答案标签，不能据此评价拒答和幻觉抑制。

## 4. 方案选择

### 4.1 方案 A：只做天池全量通用 Parent-Child

Parent 使用完整文段，Child 使用句子或语义子句。

优点是覆盖全部第三方 QA，外部泛化证据最完整。缺点是多数文段短于 500 字符，固定 `500/80` 基线对 5,693 条记录只生成一个 Chunk，容易产生天花板效应，难以证明“结构识别”的独立贡献。

### 4.2 方案 B：只做结构富集子集

只保留带编号、字段标签或明确证候/治法结构的文段。

优点是更适合观察结构感知 Parent-Child 的差异。缺点是存在选择性报告风险，不能代表天池全量数据。

### 4.3 方案 C：全量与结构子集双轨报告

采用两个互不替代的评测视图：

1. 天池全量去泄漏数据用于外部检索验证；
2. 预注册规则得到的结构富集子集用于结构感知 Parent-Child 验证。

同时在回答层使用“相同 Child 排序、只切换是否恢复 Parent”的消融，直接检验 Parent recovery 是否改善最终回答；B4 与 P 只作为端到端次要比较。

**本设计采用方案 C。**

## 6. 数据规范化与去泄漏

### 6.1 保留原始层

原始 `train.json` 只读保存，不原地修改。记录：

- 下载页面与访问日期；
- 文件大小与 SHA256；
- 数据许可页面截图或文本快照；
- 每条记录的原始 `id`；
- 原始文段、问题和答案。

### 6.2 Canonical Parent

对文段执行保守规范化：

1. Unicode NFKC；
2. 统一换行；
3. 折叠连续空白；
4. 保留文字、数字和标点，不改写医学内容。

规范化文本完全一致的记录合并为一个 `canonical_parent_id`，同时保留所有原始 ID 与 QA。

不得为了提高去重率而改写同义词、药名、方名、数字、剂量或医学术语。

### 6.3 近重复聚类

完全去重后，对文段进行近重复检测，用于阻止相似文段跨集合泄漏。候选方法为字符 n-gram MinHash 或 SimHash，最终阈值必须在开发阶段冻结，并抽样人工核查。

近重复聚类只用于切分约束，不自动合并文本或 gold 标签。

### 6.4 QA 去重

在同一个 Canonical Parent 内，按规范化后的 `question + answer` 去除完全重复 QA，同时保留原始来源映射。

相似问题但答案不同、答案范围不同或措辞不同的记录不得自动删除。

### 6.5 Gold 对齐

每个可评测 QA 至少包含：

```text
query_id
canonical_parent_id
source_record_ids
question
answer
answer_start
answer_end
alignment_method
alignment_status
```

对齐顺序：

1. 原文精确匹配；
2. 仅标点或空白差异的可逆规范化匹配；
3. 人工审核；
4. 无法唯一、可靠定位时标记为 `excluded_alignment`。

当前发现的 9 条未可靠对齐记录必须进入人工审核，不得静默修补。

### 6.6 切分原则

开发集和测试集必须按近重复 Parent 簇切分，而不是按原始 ID 或单条 QA 随机切分。

约束如下：

- 采用固定的 `20% dev / 80% test` Parent 簇切分，本阶段不训练检索或生成模型，因此不另设训练集；
- 按结构信号、Parent 长度区间和 QA 数量分层，使 dev/test 分布可比；
- 同一 Canonical Parent 的全部 QA 只能位于一个集合；
- 同一近重复簇的全部 Parent 只能位于一个集合；
- 切分脚本、随机种子、输入哈希和输出哈希必须冻结；
- 测试集冻结后不得再用于阈值、结构规则或检索参数选择。

## 7. Parent-Child 构造

### 7.1 固定切分基线

对 Canonical Parent 使用现有固定 `500/80` 递归切分，作为与项目主实验一致的基础对照。

由于绝大多数天池文段不足 500 字符，必须同时报告：

- 产生 1 个 Chunk 的 Parent 比例；
- 产生多个 Chunk 的 Parent 比例；
- 固定切分与完整 Parent 实际等价的查询比例。

### 7.2 通用 Parent-Child

不使用中医结构词典：

- Parent：完整 Canonical Parent；
- Child：标点和句界生成的句子或相同长度的通用子片段；
- 检索 Child；
- 返回时恢复唯一 Parent。

该配置用于判断 Parent recovery 本身的作用。

### 7.3 结构感知 Parent-Child

只识别原文显式存在的结构，不生成原文不存在的标题或层级。

允许使用的结构信号包括：

- 原文编号：`1.`、`1、`、`（1）`、`一、` 等；
- 原文换行与列表边界；
- 原文字段标签：证候、症状、病因、病机、治法、治则、主方、方药、组成、功效、主治、用法、用量、注意事项等；
- 连续成组出现的原文标签和值。

示例：

```text
完整疾病文段 Parent
├── 风热证：证候 Child
├── 风热证：治法 Child
├── 风热证：主方 Child
├── 寒湿证：证候 Child
├── 寒湿证：治法 Child
└── 寒湿证：主方 Child
```

若原文没有明确标签，只允许标记为通用句子或 `unknown`，不得根据模型常识补造“功效”“主治”等字段。

### 7.4 公平对照

结构感知配置必须具有一个 `P-no-structure` 对照：

- Parent 完全相同；
- Child 数量和长度分布尽量匹配；
- 检索器、Embedding、RRF、Reranker 与 Top-K 完全相同；
- 唯一差异是是否使用原文结构信号确定 Child 边界。

只有 P 相对 `P-no-structure` 的差异才能用于支持结构识别的独立贡献。

## 8. 结构富集子集

### 8.1 纳入规则

一个 Canonical Parent 只有同时满足以下条件，才进入结构富集候选集：

1. 可解析出至少两个非空 Child；
2. 至少存在一种原文显式结构信号；
3. Child 文本均可在 Parent 中定位；
4. 每个 Child 唯一恢复到一个 Parent；
5. 至少有一个有效 QA 与该 Parent 对齐。

规则必须在查看正式测试结果前冻结。

### 8.2 质量门禁

从结构候选集中分层抽取至少 200 个 Parent 人工审核，覆盖：

- 编号列表；
- 证候/治法/主方；
- 方剂组成/功效/主治；
- 用法用量/注意事项；
- 只有换行或弱结构信号的边界样本。

门禁：

- Parent 对齐正确率必须为 100%；
- Child 原文可追溯率必须为 100%；
- 结构边界精确率目标不低于 95%；
- 未达到门禁时只能修正规则并重新冻结，不能进入正式测试。

### 8.3 双轨报告

必须同时报告：

- Full：全部合格测试 QA；
- Structured：预注册结构富集子集；
- Non-Structured：其余测试 QA。

不得只报告 Structured 子集中的有利结果。

## 9. 检索实验矩阵

外部验证使用精简但能够回答研究问题的矩阵：

| 配置 | Chunk/层级 | 检索链路 | 作用 |
|---|---|---|---|
| B1 | 固定 `500/80` | BM25 | 稀疏检索基线 |
| B2 | 固定 `500/80` | Dense | 稠密检索基线 |
| B3 | 固定 `500/80` | BM25 + Dense + RRF | Hybrid 基线 |
| B4 | 固定 `500/80` | Hybrid + Reranker | 强检索基线 |
| P-generic | 通用 Child + Parent recovery | Hybrid + Reranker | 通用 Parent-Child |
| P-structure | 结构 Child + Parent recovery | Hybrid + Reranker | 完整方法 |

正式检索评测使用冻结 test 中全部满足对齐条件的 QA，不再从 test 抽取小规模检索样本。原始文件虽然名为 `train.json`，本文必须明确说明 dev/test 是本研究在 Parent 簇层面派生的评测切分，不得称为天池官方测试集。

Formal-400 的参数不得根据天池测试集重新调优。天池开发集只允许调整：

- 数据规范化规则；
- 近重复阈值；
- 结构解析规则；
- 天池语料所必需的 Child 边界参数。

检索模型和核心 Top-K 参数优先沿用 Formal-400 冻结配置。

## 10. 检索指标与统计

### 10.1 主要指标

- Parent Recall@1、Recall@5；
- Parent MRR@10；
- Parent nDCG@10；
- Gold answer span coverage@1、coverage@5；
- Parent recovery success rate；
- 返回上下文字符数与 Token 数。

天池 gold 以 Canonical Parent 和 answer span 为准，不使用自建 Evidence Group。

### 10.2 主要比较

预先指定：

1. B3 vs B1/B2：验证 Hybrid；
2. B4 vs B3：验证 Reranker；
3. P-generic vs B4：验证通用 Parent recovery；
4. P-structure vs P-generic：验证结构识别；
5. Structured 与 Non-Structured 中第 4 项差值的交互：验证增益是否集中在真实结构文本。

### 10.3 统计单位

由于一个 Parent 对应多个 QA，Bootstrap 和显著性分析必须以 Canonical Parent 或近重复簇为重采样单位，不能把 18,478 条 QA 当作完全独立样本。

主要比较报告：

- 配对差值；
- 95% Bootstrap 置信区间；
- 样本数和 Parent 数；
- Full、Structured、Non-Structured 三个视图。

## 11. Parent Recovery 回答层实验

### 11.1 样本

从冻结测试集中按 Parent 簇和结构类型分层抽取 400 条有效 QA，形成 `TCM-QG Answer-400`：

- 200 条来自 Structured；
- 200 条来自 Non-Structured；
- 每个 Canonical Parent 最多贡献固定数量问题；
- 抽样在查看回答结果前冻结。

若 Structured 测试集不足 200 条，则使用全部 Structured QA，并在 Non-Structured 中补足，总量与实际构成必须如实报告。

抽中的 400 条必须进行回答层资格审核：

- 问题能够由对应 Parent 回答；
- 标注答案不是明显截断、错字导致的错误答案或问题复述；
- answer span 能够唯一或可靠定位；
- 无效样本不得改写答案，只能标记排除，并按预先冻结的候补顺序替换；
- 至少随机复核 40 条，记录一致性和分歧处理。

Structured 与 Non-Structured 必须分别报告。由于 Answer-400 对 Structured 进行了目的性过采样，50/50 汇总结果不能表述为天池全量数据的总体性能估计。

### 11.2 配置

保持相同生成模型、提示词、温度、Token 上限和引用格式：

- P-generic-child-only：使用通用 Parent-Child 的检索排序，只提供命中的 Child；
- P-generic-parent：保持相同排序，将命中 Child 恢复为 Parent；
- P-structure-child-only：使用结构感知 Parent-Child 的检索排序，只提供命中的 Child；
- P-structure-parent：保持相同排序，将命中 Child 恢复为 Parent；
- B4-answer：使用 B4 返回的固定 Chunk，作为端到端强基线。

B0 不纳入本阶段主要矩阵。若后续为补充 RQ1 单独加入，必须另行预注册，且不能影响 Parent-Child 的主要比较。

回答层主要比较为：

1. P-generic-child-only vs P-generic-parent；
2. P-structure-child-only vs P-structure-parent。

这两个比较保持检索结果不变，只改变生成上下文是否恢复 Parent，能够隔离 Parent recovery。B4-answer vs P-structure-parent 仅用于评价完整系统，不用于单独归因 Parent recovery。

### 11.3 回答指标

- Exact Match；
- 中文字符级或 Token F1；
- Gold answer 支持率；
- 引用 Parent 正确率；
- 引用证据覆盖率；
- 回答内容可由返回证据支持的比例；
- 平均输入上下文长度、输出 Token 与延迟。

本阶段只评价“是否得到源文段支持”，不评价临床建议正确性或现实医疗有效性。

## 12. 判定规则

### 12.1 支持结构感知贡献

只有满足以下证据链，才可以宣称结构识别产生独立贡献：

1. P-structure 相对 P-generic 的主要指标差值为正；
2. 配对置信区间能够排除 0，或多个主要指标呈一致方向；
3. Structured 子集增益大于 Non-Structured；
4. 结构解析质量通过人工门禁。

### 12.2 只支持通用 Parent recovery

若 P-generic 优于 B4，但 P-structure 与 P-generic 无可靠差异，则结论只能是：

> 通用 Parent recovery 在该外部数据上有效，尚无证据证明中医结构规则产生额外收益。

### 12.3 只支持回答层收益

若检索排序差异不明显，但 P 在回答 F1、证据支持率或引用覆盖率上改善，则可以表述为：

> Parent recovery 的主要价值体现在提供更完整的生成上下文，而非改变检索排序。

该结论必须优先来自 `child-only` 与 `parent` 的同排序消融。B4 与 P 的端到端差异同时包含切分和检索差异，不能单独用于证明 Parent recovery。

### 12.4 阴性结果

若 B4、P-generic 和 P-structure 无可靠差异，必须报告天池短文导致的天花板效应，不得用 Structured 子集中的个别指标替代全量结论。

## 16. 论文中的推荐表述

可表述为：

> 本文首先在自建并经人工审核的古籍证据型评测集上开展受控主实验，随后引入天池公开的中医文献人工问答数据进行独立外部验证。针对公开数据中存在的重复文段，本文在文段簇层面执行去重和数据切分，并分别报告全量数据与显式结构子集的结果，以区分通用 Parent recovery 和中医结构识别的贡献。

不能表述为：

- 天池官方提供了 Parent-Child 检索基准；
- 天池数据天然包含完整的篇章层级；
- 结构标签由天池官方标注；
- 只要恢复完整文段就证明结构感知方法有效；
- 公开数据上的结果可以替代 Formal-400 古籍主实验。

## 17. 最终研究证据链

完成本设计后，论文证据链为：

1. **Formal-400：** 受控古籍主实验，验证古籍场景中的基线、消融和检索排序；
2. **TCM-QG Full：** 第三方人工 QA 外部验证，降低自建题目偏差；
3. **TCM-QG Structured：** 真实结构信号子集，隔离结构识别的贡献；
4. **TCM-QG Answer-400：** 同排序 Child-only/Parent 消融，直接验证 Parent recovery 对证据完整性和忠实度的作用，并用 B4/P 补充端到端比较。

这一组合允许论文区分“通用层级检索有效”“中医结构规则有效”和“完整上下文改善回答”三个不同命题，避免把它们混写为一个未经隔离的结论。
