# TCM-Flow V1.6 Public TCM-QG Formal Experiment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在 V1.6 独立目录中完成公开 TCM-QG 的正式 BGE/Reranker/LLM 实验，判断 Parent-Child 是否能支撑论文中的“结构感知混合检索增强”创新主张。

**Architecture:** 本计划从已完成的 V1.6 offline proxy 结果出发，但不把 proxy 当作正式结论。正式实验恢复 BGE-M3 dense、BGE reranker 和真实 LLM 回答层；检索在完整 dev/test split 上执行，回答层先做成本预估和 dev 门禁，再对 test 单次冻结运行。论文主张分两级冻结：强主张要求 `P > B4`，保底主张只允许写 `P > P-no-parent` 的 parent context 消融贡献。

**Tech Stack:** Python 3.10、Pydantic 2、JSONL/CSV、PyYAML、NumPy、jieba BM25、BGE-M3、BGE Reranker、LangChain `ChatOpenAI`、OpenAI-compatible API、paired bootstrap、人工盲审、`unittest`。

---

## 1. 为什么需要正式实验

V1.6 已完成的 proxy run 只说明“证据是否被检索出来”，不是正式 LLM RAG 结果。关键观察是：

```text
P vs P-no-parent: positive
P vs B4: negative under lexical proxy
public_tcm_qg_success=false
```

因此正式实验的任务不是预设 Parent-Child 一定成功，而是回答三个论文问题：

1. Parent context 是否在真实 BGE/Reranker/LLM 下稳定优于 child-only？
2. Parent-Child 是否能整体超过普通 chunk RAG baseline `B4`？
3. 如果 `P` 没有超过 `B4`，论文是否仍能保留“结构感知证据组织”这个较窄创新点？

本计划不得把结果写成“多源异构融合显著增强”，除非后续真的引入多个异构来源并在实验中验证。当前 V1.6 正式实验最多验证：

```text
结构感知 Parent-Child 证据组织 + Hybrid/Reranker + LLM 回答
```

更稳妥的论文表述：

```text
提出一种面向中医文献问答的结构感知混合检索增强机制。该机制在 BM25、Dense、RRF 与重排序基础上，引入 Parent-Child 文档组织与证据回收策略，使系统在保持细粒度检索能力的同时，为回答生成提供更完整的上层语义上下文，从而提升证据覆盖度与回答可追溯性。
```

## 3. 正式实验假设

### 3.1 强主张门槛

只有同时满足以下条件，论文才可以写：

```text
Parent-Child RAG 在公开 TCM-QG 上相较普通 chunk RAG 取得稳定提升。
```

成功条件：

```text
P_vs_B4_char_f1_delta > 0
P_vs_B4_char_f1_95ci_lower > 0
P_vs_B4_citation_recall_delta >= 0
P_vs_B4_unsupported_answer_rate_delta <= 0
P_vs_P_no_parent_char_f1_delta > 0
P_vs_P_no_parent_char_f1_95ci_lower > 0
P_vs_P_no_parent_citation_recall_delta > 0
P_vs_P_no_parent_citation_recall_95ci_lower > 0
```

### 3.2 保底主张门槛

如果 `P > P-no-parent` 成立，但 `P > B4` 不成立，只能写：

```text
Parent context 对 child retrieval 的证据展开和回答可追溯性有贡献，但当前整体方法未超过普通 chunk RAG baseline。
```

保底成功条件：

```text
P_vs_P_no_parent_char_f1_delta > 0
P_vs_P_no_parent_char_f1_95ci_lower > 0
P_vs_P_no_parent_citation_recall_delta > 0
P_vs_P_no_parent_citation_recall_95ci_lower > 0
```

### 3.3 失败解释门槛

如果 `P > P-no-parent` 也不成立，必须写失败分析，不得继续包装为 Parent-Child 成功：

```text
Parent-Child 在当前公开 TCM-QG 设置下未显示稳定收益，后续需要改进 child ranking、parent 长度控制或证据压缩策略。
```

## 4. 固定实验矩阵

### 4.1 检索矩阵

正式检索至少运行以下 7 个配置：

| Config ID | 论文角色 | Chunk/上下文 | 检索方式 |
| --- | --- | --- | --- |
| `b1-public-bm25` | B1 | B4 `500/80` chunk | BM25 |
| `b2-public-dense` | B2 | B4 `500/80` chunk | BGE-M3 Dense |
| `b3-public-hybrid` | B3 | B4 `500/80` chunk | BM25 + Dense + RRF |
| `b4-public-hybrid-rerank` | B4 | B4 `500/80` chunk | Hybrid + BGE Reranker |
| `p-public-hybrid-rerank` | P | Child 检索 + Parent context | Hybrid + BGE Reranker |
| `p-public-no-parent` | Parent 消融 | 与 P 相同 child 排序 + child context | Hybrid + BGE Reranker |
| `p-public-no-reranker` | Reranker 消融 | Child 检索 + Parent context | Hybrid only |

可选诊断配置只在主结果失败后执行，不进入主成功门槛：

```text
p-public-no-bm25
p-public-no-dense
p-public-short-parent
p-public-titleless-parent
```

### 4.2 回答矩阵

正式回答层只运行：

```text
B0
B4
P
P-no-parent
```

`B1/B2/B3/p-no-reranker` 只用于检索诊断，不进入 LLM 回答主矩阵，避免成本失控和论文叙述发散。

回答参数：

```text
temperature=0
repeats=1
max_tokens=256
context_top_k=5
test_policy=single_frozen_run
dev_usage=prompt_and_cost_gate_only
```

如果导师要求稳定性分析，再对 dev sample 运行 `repeats=3`，不得直接扩大 test 重复次数。

## 15. 论文写法边界

### Strong success

可以写：

```text
公开 TCM-QG 正式实验显示，结构感知 Parent-Child 混合检索增强方法在回答字符 F1、引用召回和人工证据支持指标上相较普通 chunk RAG 取得稳定提升。
```

### Parent-ablation-only

只能写：

```text
Parent context 相较 child-only 消融显著提升了证据覆盖与回答可追溯性，但当前完整 Parent-Child 方法未超过普通 chunk RAG baseline。因此本文将该结果解释为结构化证据组织的局部贡献，而非整体方法最优。
```

### Failed

必须写：

```text
Parent-Child 在公开 TCM-QG 正式实验中未显示稳定收益，本文保留该方法作为系统设计探索，并将失败原因归因于 child 排序、parent 长度噪声或证据压缩不足。
```

任何情况下都不得写：

```text
未被实验支持的“多来源异构融合带来显著整体提升”强表述
```

除非后续实验真的引入多来源、多结构的数据，并完成相应对照。
