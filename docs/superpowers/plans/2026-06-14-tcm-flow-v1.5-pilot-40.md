# TCM-Flow V1.5 Pilot-40 Retrieval Experiment Implementation Plan

> **For Codex:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task.

**Goal:** 在已通过的 Quality Gate、C0-C4 真实索引和 Smoke-10 人工门禁之上，建立并冻结 40 条 evidence-first 检索试验集，完成固定的 8 组 Pilot 检索矩阵，产出可追溯、可恢复、可用于决定是否进入正式 400 题评测的阶段报告。

**Architecture:** 所有新增实现继续位于离线实验目录 `experiments/rag_v1_5`，不修改在线 `app/rag`。数据侧用“Evidence Group -> 本地草稿 JSONL -> 双轮人工复核 CSV -> 冻结 JSONL/Manifest”形成审计链；运行侧用“预加载索引与模型 -> 单题检索与分阶段计时 -> 可恢复矩阵运行 -> 聚合报告”形成结果链。完整问题、原文片段、人工评论和逐题运行结果只保存在已被 Git 忽略的 `data/rag_v1_5`；仓库只提交代码、测试、配置、去内容化 Manifest、实验日志和阶段报告。

**Tech Stack:** Python 3.11、Pydantic v2、PyYAML、NumPy、rank-bm25、FlagEmbedding、Transformers、`unittest`、PowerShell、JSONL、UTF-8 BOM CSV、SHA256。

## 1. Pilot-40 固定实验设计

### 1.1 问题配额

每个“书籍 × 问题类型”单元恰好 4 条，共 40 条：

| 问题类型 | 《伤寒论》 | 《金匮要略》 | 合计 |
| --- | ---: | ---: | ---: |
| `single_clause_fact` | 4 | 4 | 8 |
| `formula_composition_or_use` | 4 | 4 | 8 |
| `source_location` | 4 | 4 | 8 |
| `multi_evidence` | 4 | 4 | 8 |
| `unanswerable` | 4 | 4 | 8 |
| **合计** | **20** | **20** | **40** |

固定计数：

- 可回答题：32。
- 无答案题：8。
- `book_scope` 只允许 `shang_han_lun` 或 `jin_gui_yao_lue`，Pilot-40 不使用 `both`。
- `multi_evidence` 每题至少有 2 个 gold clause，且来自同一本书。
- `unanswerable` 对当前两部冻结语料均无可支持答案；`book_scope` 仅表示题目归属配额，不表示只在该书内搜索。

### 1.2 固定 8 组矩阵

矩阵只改变 Chunk 策略或检索模式，不改变模型、索引、候选数、RRF、Reranker 或问题集：

| config_id | Chunk | Mode | 对应研究问题 |
| --- | --- | --- | --- |
| `c0-hybrid-rerank` | C0 | Hybrid + Reranker | 通用 `500/80` 锚点 |
| `c1-hybrid-rerank` | C1 | Hybrid + Reranker | 短通用 Chunk |
| `c2-hybrid-rerank` | C2 | Hybrid + Reranker | clause Chunk |
| `c3-hybrid-rerank` | C3 | Hybrid + Reranker | structure-aware Chunk |
| `c4-bm25` | C4 | BM25 | 单路稀疏检索 |
| `c4-dense` | C4 | Dense | 单路向量检索 |
| `c4-hybrid` | C4 | Hybrid | 融合增益 |
| `c4-hybrid-rerank` | C4 | Hybrid + Reranker | Parent-Child 完整管线 |

该矩阵支持对 RQ2-RQ5 做 Pilot 级检查。无答案题只记录 Top-1/Top-5 分数分布，Pilot 阶段不设拒答阈值，也不计入 Recall、Hit、MRR、nDCG。

### 1.3 人工门禁

第一轮审核全部 40 条。第二轮固定抽 10 条，按“2 本书 × 5 问题类型”每层抽 1 条，种子固定为 `20260614`。

第二轮是独立轮次复核，不强制声称为独立专家复核。同一审核者可以在不同时间完成两轮，但报告必须明确这一限制。

最终冻结条件：

```text
question_count=40
answerable_count=32
unanswerable_count=8
first_review_pass_count=40
first_review_pending_count=0
second_review_required_count=10
second_review_pass_count=10
unresolved_gold_id_errors=0
unresolved_support_errors=0
unresolved_relevance_errors=0
unresolved_clinical_scope_errors=0
```

## Task 4: Evidence-first 编写、审核并冻结 40 条问题

**Files:**

- Local only: `data/rag_v1_5/evaluation/pilot-evidence-groups.jsonl`
- Local only: `data/rag_v1_5/evaluation/pilot-40-draft.jsonl`

该 Task 包含人工检查点。不要在用户未完成 CSV 审核时假设审核通过。

### Step 1: 先填 gold，再写问题

每个 answerable group 按以下顺序构造：

1. 固定 `question_id`，格式为 `pilot-<book>-<type>-<01..04>`。
2. 填 `split=pilot`、`evidence_group_id`、`question_version=1`。
3. 填 `gold_evidence_ids`、`gold_clause_ids`。
4. 从 gold Evidence 原文中逐字摘取最短充分 `support_spans`。
5. 填 `graded_relevance`：直接回答核心事实的 ID 为 `2`，仅提供辅助定位或第二证据的 ID 为 `1`。
6. 写 `reference_answer`，只做古籍文本事实转述。
7. 最后写自然语言 `question`，不得出现内部 ID。
8. 初始 `review_status=draft`。

禁止：

- 患者个案诊断。
- 辨证论治建议。
- 剂量、加减、服药或停药建议。
- 疗效、安全性或现代临床结论。
- 原文未支持的因果推断。
- 用检索结果倒推 gold。
- 调用 LLM API 批量生成问题或答案。

### Step 2: 构造 8 条无答案题

每条无答案题必须：

- `gold_evidence_ids=[]`
- `gold_clause_ids=[]`

## Task 6: 增加可复用索引和分阶段计时的检索入口

**Files:**

- Modify: `experiments/rag_v1_5/retrieval.py`
- Modify: `tests/rag_v1_5/test_retrieval.py`
- Modify: `tests/rag_v1_5/test_dataset.py`

当前 `retrieve()` 每题都会 `load_index()`，且只返回 hits，无法为矩阵提供可比的阶段耗时。保留现有 API，增加运行器专用的预加载入口。

### Step 1: 先写失败测试

新增测试：

1. `retrieve_loaded()` 接收已加载 `LoadedIndex`，不会再次读索引文件。
2. 四种 mode 均返回固定 latency key：
   `bm25_ms/dense_ms/rrf_ms/reranker_ms/total_ms/returned_context_chars`。
3. 未启用阶段的耗时为 `0.0`，不是缺失或 `null`。
4. 每个耗时非负，`total_ms` 不小于已启用阶段之和的容差下界。
5. hits 排序与旧 `retrieve()` 完全一致。
6. C4 hits 保留 `retrieval_parent_id/context_text/clause_ids/source_evidence_ids`。
7. Smoke-10 旧入口和测试不回归。
8. 默认 `result_top_k=None` 时保持现有 Top 5 行为；Pilot 传 `result_top_k=10` 时四种 mode 都返回最多 10 条，保证 Recall@10 有真实排名深度。

### Step 2: 增加结果对象和预加载入口

在 `retrieval.py` 增加：

```python
@dataclass
class RetrievalResult:
    hits: list[RetrievalHit]
    latency: dict[str, float | int]


def retrieve_loaded(
    query: str,
    *,
    index: LoadedIndex,
    mode: RetrievalMode,
    config: dict,
    dense_encoder: DenseEncoder | None = None,
    reranker_scorer: RerankerScorer | None = None,
    result_top_k: int | None = None,
) -> RetrievalResult:
    ...
```

计时边界：

- `bm25_ms`: tokenize + BM25 score + 排序。
- `dense_ms`: query encode + dense score + 排序。
- `rrf_ms`: RRF 融合 + 截断候选。
- `reranker_ms`: pair 构造 + 模型打分 + 排序。
- `total_ms`: 从查询校验开始到 Top K hits 完成。
- `returned_context_chars`: 面向上下文的前 5 条 hits 的 `context_text` 字符数之和；即使 Pilot 为 Recall@10 保存 10 条排名，也不把后 5 条计入上下文长度。

`result_top_k` 的规则：

- `None` 时使用当前 `config["reranker"]["top_k"]`，所以现有 search/Smoke 仍为 Top 5。
- Pilot runner 固定传 `max(config["evaluation"]["top_ks"])`，当前为 10。
- BM25、Dense、Hybrid 和 Hybrid+Reranker 都按该深度返回。
- Reranker 仍只在固定 `candidate_k=40` 内排序；评测深度从 5 扩到 10 不改变候选集合、模型或分数。

模型加载和索引加载不计入单题 latency；运行器单独记录 `index_load_ms`、`embedding_load_ms`、`reranker_load_ms`。

旧 `retrieve()` 改为：

```python
index = load_index(indexes_dir / strategy)
return retrieve_loaded(
    query,
    index=index,
    mode=mode,
    config=config,
    dense_encoder=dense_encoder,
    reranker_scorer=reranker_scorer,
    result_top_k=None,
).hits
```

## 2. 执行过程中的停止条件

出现以下任一情况，立即停止，不继续跑矩阵：

- Quality Gate、Chunk、Index、Model、config 或 Smoke hash 与冻结 Manifest 不一致。
- Pilot 配额、gold、support、review 或 privacy 契约失败。
- 运行时 CUDA 不可用。
- 真实运行中出现非偶发的 parser/gold/traceability 错误。
- 为提高 Pilot 指标而修改问题、gold、参数或矩阵。
- Pilot Evidence/Clause 被提前用于构造正式 test-200/400。

偶发进程中断不属于实验失败；使用同一 matrix 目录 resume，并在日志中记录中断和恢复。
